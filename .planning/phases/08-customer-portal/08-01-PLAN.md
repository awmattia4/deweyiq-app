**Create `src/lib/db/schema/portal-messages.ts`:**
- Table: `portal_messages` with columns: id (uuid PK), org_id (FK orgs cascade), customer_id (FK customers cascade), service_request_id (uuid FK service_requests set null, nullable — when set, this message belongs to a specific request thread), sender_role (text, 'customer' | 'office'), sender_name (text), body (text nullable), photo_path (text nullable for single photo attachment), read_by_office_at (timestamptz nullable), read_by_customer_at (timestamptz nullable), created_at (timestamptz default now)
- Indexes: customer_id, org_id, service_request_id
- RLS: Two policies:
  1. `portal_messages_office_policy` — for "all", using: `org_id = JWT org_id AND user_role IN ('owner', 'office')`
  2. `portal_messages_customer_policy` — for "all", using: `org_id = JWT org_id AND user_role = 'customer' AND customer_id IN (SELECT id FROM customers WHERE email = auth.email() AND org_id = JWT org_id)`
- Note: Use the exact `sql` template literal pattern from existing schema files (e.g., service-visits.ts). Use `(select auth.jwt() ->> 'org_id')::uuid` syntax. Use `(select auth.email())` for email. See 08-RESEARCH.md Pattern 4 for exact SQL.

**Create `src/lib/db/schema/service-requests.ts`:**
- Table: `service_requests` with columns: id (uuid PK), org_id (FK orgs cascade), customer_id (FK customers cascade), pool_id (uuid FK pools set null), work_order_id (uuid nullable, no FK to avoid circular deps), category (text, 'green_pool' | 'opening_closing' | 'repair' | 'cleaning' | 'chemical' | 'other'), description (text), is_urgent (boolean default false), photo_paths (jsonb string[]), preferred_date (text for YYYY-MM-DD), preferred_time_window (text, 'morning' | 'afternoon' | 'anytime'), status (text default 'submitted', values: submitted | reviewed | scheduled | complete | declined), office_notes (text nullable), created_at (timestamptz default now), updated_at (timestamptz default now)
- Indexes: org_id, customer_id, status
- RLS: Two policies (same pattern as portal_messages):
  1. Office policy: SELECT/INSERT/UPDATE/DELETE for owner+office (use `for: "all"`)
  2. Customer policy: SELECT + INSERT for customer role only, scoped by email lookup. Use separate SELECT and INSERT policies for customer — SELECT uses `using`, INSERT uses `withCheck`, both with the same customer_id IN subquery. Customer should NOT be able to UPDATE or DELETE requests.

**Create `src/actions/portal-auth.ts`:**

`sendMagicLink(email: string)`:
- Uses `supabase.auth.signInWithOtp({ email, options: { shouldCreateUser: false, emailRedirectTo } })` — CRITICAL: `shouldCreateUser: false` prevents rogue account creation (see Pitfall 5 in research).
- `emailRedirectTo` points to `/auth/portal-callback` (use `process.env.NEXT_PUBLIC_APP_URL` as base).
- Always returns `{ success: true }` regardless of whether email exists (prevents email enumeration). Log non-rate-limit errors to console.

`getCustomerOrgs(userId: string, email: string)`:
- Uses `adminDb` to query: find all profiles where `email = email AND role = 'customer'`. For each profile, join with `orgs` to get `org.id, org.name, org.logo_url, org.slug`. Also LEFT JOIN `org_settings` to get `brand_color`.
- Returns array of `{ orgId, orgName, logoUrl, slug, brandColor }`.
- This is the multi-org company picker data source.

`switchOrg(newOrgId: string)`:
- Verify the customer actually has a profile in the target org (query profiles by userId + orgId).
- Use `adminDb` to call `supabase.auth.admin.updateUserById(userId, { app_metadata: { org_id: newOrgId } })` to update the active org in the JWT.
- Return `{ success: true }` — client will call `supabase.auth.refreshSession()` and do a hard navigation.

**Create `src/actions/portal-data.ts`:**

`resolveCustomerId(orgId: string, email: string)`:
- Uses `adminDb` to find `customers.id` WHERE `email = email AND org_id = orgId LIMIT 1`.
- Returns `customerId: string | null`.
- This is the canonical way to map a logged-in customer to their `customers` row. All portal data queries use this.

`getOrgBranding(orgId: string)`:
- Uses `adminDb` to SELECT from `orgs` JOIN `org_settings` WHERE `orgs.id = orgId`.
- Returns `{ name, logoUrl, slug, brandColor, portalWelcomeMessage, faviconUrl }`. For `faviconUrl`: if `org_settings.favicon_path` exists, generate a signed URL with 24-hour expiry; otherwise null.
- Used by portal layout to inject branding.

**Create `src/app/auth/portal-callback/route.ts`:**
- GET handler that extracts `token_hash` and `type` from search params.
- Calls `supabase.auth.verifyOtp({ token_hash, type })`.
- On success: redirect to `/portal`.
- On failure: redirect to `/portal/login?error=invalid_link`.
- Pattern matches the existing `/auth/callback/route.ts` but simpler — no OAuth, no recovery, just OTP.
