**Add to `src/actions/portal-data.ts` (created in Plan 01):**

`getServiceHistory(orgId: string, customerId: string)`:
- Uses `adminDb` (NOT withRls — customer role can't SELECT existing tables without RLS changes; use adminDb with explicit customerId filter per research recommendation).
- Query: SELECT service_visits.*, pools.name as pool_name, pools.pool_type, pools.sanitizer_type, profiles.full_name as tech_name FROM service_visits LEFT JOIN pools ON service_visits.pool_id = pools.id LEFT JOIN profiles ON service_visits.tech_id = profiles.id WHERE service_visits.customer_id = customerId AND service_visits.org_id = orgId AND service_visits.status IN ('complete', 'skipped') ORDER BY service_visits.visited_at DESC
- Also query `pools` for this customer: SELECT id, name, pool_type, sanitizer_type FROM pools WHERE customer_id = customerId AND org_id = orgId
- Return `{ pools: Pool[], visits: VisitWithMeta[] }` where VisitWithMeta includes pool_name, pool_type, sanitizer_type, tech_name, and all service_visit fields.
- **CRITICAL (from MEMORY.md):** Include `chemistry_readings` field — do NOT drop it during object mapping. The Drizzle relational query fetches all columns but manual mapping can silently drop fields.

`getVisitPhotos(orgId: string, customerId: string, poolId?: string)`:
- Uses `adminDb` to query visit_photos for visits belonging to this customer.
- Join: visit_photos -> service_visits (via visit_id or org scoping) -> filter by customer_id.
- Actually, visit photos are stored as `photo_urls: jsonb string[]` in service_visits AND in the visit_photos table. Query both sources.
- For each photo path, generate a signed URL using `adminDb` storage: `supabase.storage.from('company-assets').createSignedUrl(path, 3600)` (1-hour expiry per research recommendation).
- Return `{ photos: { url: string, visitDate: string, poolName: string }[] }` sorted newest first.
- If `poolId` is provided, filter to only that pool's visit photos.
