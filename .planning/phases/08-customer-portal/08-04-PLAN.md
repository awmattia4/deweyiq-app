**Create `src/actions/service-requests.ts`:**

`submitServiceRequest(orgId: string, customerId: string, data: ServiceRequestInput)`:
- Input: `{ poolId, category, description, isUrgent, photoDataUrls: string[], preferredDate, preferredTimeWindow }`.
- Uses `adminDb` to INSERT into service_requests with status 'submitted'.
- Photo handling: For each photoDataUrl, compress with `browser-image-compression` pattern (but since this is a server action, photos arrive as base64 data URLs from the client). Actually, photo upload should happen client-side via signed URLs (matches existing pattern). The action receives `photoPaths: string[]` (already-uploaded Storage paths).
- Revised: Accept `photoPaths: string[]` (paths already uploaded to Storage). Store in `service_requests.photo_paths`.
- If `isUrgent` is true, send an email notification to the office (use Resend, same pattern as alert notifications). Use the notification template system if a template exists for 'service_request_submitted'.
- Returns `{ id: string, success: true }`.

`createRequestPhotoUploadUrl(orgId: string, customerId: string, filename: string)`:
- Generates a signed upload URL: `supabase.storage.from('company-assets').createSignedUploadUrl(`${orgId}/portal/requests/${Date.now()}-${filename}`)`.
- Returns `{ signedUrl, path }`.
- Customer uploads directly to the signed URL from the client.

`getCustomerRequests(orgId: string, customerId: string)`:
- Uses `adminDb`. SELECT service_requests.*, pools.name as pool_name FROM service_requests LEFT JOIN pools ON pool_id = pools.id WHERE customer_id = customerId AND org_id = orgId ORDER BY created_at DESC.
- Returns `ServiceRequest[]`.

`getOfficeRequests(orgId: string)`:
- Uses `withRls` (office/owner role). SELECT service_requests.*, customers.full_name as customer_name, pools.name as pool_name FROM service_requests LEFT JOIN customers ON customer_id = customers.id LEFT JOIN pools ON pool_id = pools.id WHERE org_id = orgId ORDER BY is_urgent DESC, created_at DESC.
- Urgent requests sort to top.
- Returns with customer name and pool name.

`reviewRequest(requestId: string, data: { status: 'reviewed' | 'declined', officeNotes?: string })`:
- Uses `withRls`. UPDATE service_requests SET status, office_notes, updated_at.
- Owner or office role required.

`createWoFromRequest(requestId: string)`:
- Uses `withRls` + `adminDb` for cross-table operations.
- Read the service request to get customer_id, pool_id, category, description, is_urgent.
- Create a work_order: title from "Service Request: {category}", description from request description, category mapped appropriately, priority = is_urgent ? 'high' : 'normal', status = 'draft', target_date = preferred_date.
- Update service_request: set work_order_id to the new WO id, set status to 'reviewed'.
- Returns the new work_order id.
- This integrates with the existing WO system from Phase 6.

**Add to `src/actions/portal-data.ts`:**

`getCustomerPools(orgId: string, customerId: string)`:
- Uses `adminDb`. SELECT id, name, pool_type FROM pools WHERE customer_id = customerId AND org_id = orgId.
- Used by the request form to populate pool selection dropdown.
