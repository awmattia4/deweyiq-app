**Create src/components/work-orders/flag-issue-sheet.tsx** — "use client" component:
- Bottom sheet (shadcn Sheet, side="bottom", max-w-lg mx-auto for mobile) triggered by a "Flag Issue" button.
- Props: customerId, poolId, visitId (from current stop context), onClose callback.
- Optimized for 10-second completion — large tap targets (44px min), minimal fields:

1. **Category picker:** 7 pill-style buttons in a flex-wrap row: Pump, Filter, Heater, Plumbing/Leak, Surface, Electrical, Other. Pre-selected = none (required). Large pills, easy one-handed tap.

2. **Severity picker:** 3 pill buttons in a row: Routine (gray), Urgent (amber), Emergency (red). Default = "routine".

3. **Note field:** Single-line textarea with placeholder "Describe the issue..." and a microphone icon hint for system voice dictation (same pattern as Phase 3 NotesField — system keyboard dictation, NOT Web Speech API).

4. **Photo capture:** Reuse the existing photo capture pattern from Phase 3 (camera button → browser file input with capture="environment", compress with browser-image-compression to webp, display thumbnails). Max 3 photos. Photos are uploaded to Supabase Storage bucket `work-order-photos` at path `{orgId}/work-orders/{woId}/{filename}.webp` using the existing `getSignedUploadUrl` pattern from storage.ts. Since woId doesn't exist yet at photo time, store photos in a temp Dexie queue or upload after WO creation.

5. **Submit button:** "Flag Issue" — calls `createWorkOrder` with:
   - status: 'draft'
   - customerId, poolId from props
   - title: auto-generated from category (e.g., "Pump Issue — Flagged by Tech")
   - category: selected category
   - severity: selected severity
   - description: note text
   - flaggedByTechId: current user id (from auth)
   - flaggedFromVisitId: visitId from props
   After WO creation, upload photos to `work-order-photos/{orgId}/work-orders/{newWoId}/`. Then close sheet + show toast "Issue flagged".

6. **Offline consideration:** If offline, queue the createWorkOrder call via enqueueWrite pattern. Photos go to Dexie photo queue for later upload (same as visit photos offline flow).

**Create src/components/work-orders/wo-tech-completion.tsx** — "use client" component:
- This is a bottom sheet or page section shown when a tech opens a WO that is assigned to them and status is 'scheduled' or 'in_progress'.
- Props: workOrder (full WO object with line items).

**Arrival flow (scheduled → in_progress):**
- "Mark Arrived" button (large, prominent, green).
- On tap: calls `updateWorkOrderStatus(woId, 'in_progress')`.
- Transition recorded in activity_log.

**Completion flow (in_progress → complete):**
- Shows work order details: title, description, line items.
- **Add parts at completion:** If parts were used that weren't on the original WO, tech can add them. Simple form: description + quantity + "Save to catalog?" checkbox. This calls a new `addLineItemToWorkOrder` action (add to work-orders.ts if not there).
- **Log actual hours:** For labor line items with labor_type='hourly', show an input to log actual_hours. Calls `updateLineItemActualHours` action.
- **Completion photos:** Photo capture (reuse Phase 3 pattern) — "Add completion photos" button. Photos uploaded to `work-order-photos/{orgId}/work-orders/{woId}/completion/`.
- **Completion notes:** Textarea for describing what was done.
- **"Mark Complete" button:** Calls `updateWorkOrderStatus(woId, 'complete', { completionNotes, completionPhotoPaths })`.
- On completion: toast "Work order completed", auto-notification to customer if org_settings.wo_notify_customer_on_complete is true (this notification logic should be in the updateWorkOrderStatus action — send via existing alert/notification pattern).

**Add to work-orders.ts actions (if not already):**
- `addLineItemToWorkOrder(woId, lineItem)` — Inserts a new line item. Tech role allowed per RLS.
- `updateLineItemActualHours(lineItemId, hours)` — Updates actual_hours on a line item.
- `getAssignedWorkOrders()` — Fetches WOs assigned to the current tech user (for tech's "My Work Orders" view). Filter by assigned_tech_id = current user, status in ['scheduled', 'in_progress'].

**Tech access point:** The tech views assigned WOs from their routes page or a dedicated section. For now, create the completion component — the routing integration (how tech navigates to a WO) will be naturally wired when the tech taps on a WO from their assigned list or from the stop workflow.
  </action>
