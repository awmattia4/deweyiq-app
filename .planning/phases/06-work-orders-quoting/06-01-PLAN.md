**Create src/actions/work-orders.ts** with "use server" directive. Implement:

1. `getWorkOrders(filters?)` — Fetches WO list with LEFT JOIN to customers, pools, profiles (for tech name). Filters by status[], priority, techId, customerId. Orders by priority (emergency=0, high=1, normal=2, low=3) then created_at desc. Uses `withRls`. Returns array of WO summary objects. See 06-RESEARCH.md WO List Query pattern.

2. `getWorkOrder(id)` — Fetches single WO with all relations: customer, pool, assigned tech, created by, flagged by tech, line items (ordered by sort_order), quotes (ordered by version desc). Uses `withRls` with relational query or LEFT JOINs.

3. `createWorkOrder(data)` — Creates a WO record. Accepts: customerId, poolId, title, description, category, priority, severity (optional — for tech flags), flaggedByTechId (optional), flaggedFromVisitId (optional), templateId (optional). Sets org_id from JWT claims. Initializes activity_log with `[{ type: 'created', at: new Date().toISOString(), by_id: userId, note: null }]`. If templateId provided, also creates line items from the template's line_items_snapshot. Uses `withRls`. Returns the created WO id. Calls `revalidatePath('/work-orders')`.

4. `updateWorkOrder(id, data)` — Updates WO fields (title, description, category, priority, assignedTechId, targetDate). Appends to activity_log. Uses `withRls`. Calls `revalidatePath`.

5. `updateWorkOrderStatus(id, newStatus, extra?)` — Transitions WO status. Validates allowed transitions: draft->quoted, draft->approved (skip-quote), quoted->approved, approved->scheduled, scheduled->in_progress, in_progress->complete, complete->invoiced, any-except-complete/invoiced->cancelled. For 'scheduled': requires assignedTechId + targetDate in extra. For 'complete': accepts completionNotes, completionPhotoPaths in extra; sets completed_at. For 'cancelled': requires cancelReason in extra; sets cancelled_at, cancelled_by_id. Appends activity_log event. Uses `withRls`. Calls `revalidatePath`.

6. `createFollowUpWorkOrder(parentWoId)` — Fetches parent WO, creates a new draft WO pre-filled with same customer, pool, category, references parent_wo_id. Appends 'follow_up_created' to parent's activity_log. Uses `withRls`.

7. Helper: `appendActivityEvent(db, workOrderId, event)` — Appends a JSONB event to activity_log using `COALESCE(activity_log, '[]'::jsonb) || ${JSON.stringify([event])}::jsonb` SQL pattern.

**Create src/actions/parts-catalog.ts** with "use server" directive. Implement:

1. `getCatalogItems(search?, category?)` — Fetches active catalog items for the org. Optional ILIKE search on name, optional category filter. Uses `withRls`. Returns array.

2. `addCatalogItem(data)` — Creates a catalog item. Accepts name, description, category, sku, defaultCostPrice, defaultSellPrice, defaultUnit, isLabor. Uses `withRls`.

3. `updateCatalogItem(id, data)` — Updates a catalog item. Uses `withRls`.

4. `deleteCatalogItem(id)` — Soft-delete: sets is_active=false. Uses `withRls`.

5. `getWoTemplates()` — Fetches active WO templates for the org. Uses `withRls`.

6. `createWoTemplate(data)` — Creates a template with name, description, category, defaultPriority, lineItemsSnapshot. Uses `withRls`.

7. `deleteWoTemplate(id)` — Soft-delete: sets is_active=false. Uses `withRls`.

All actions follow established patterns: getRlsToken() guard, withRls(token, async (db) => ...), revalidatePath on mutations. Import tables from @/lib/db/schema. Use `eq`, `and`, `ilike`, `inArray`, `desc`, `sql` from drizzle-orm as needed.
