**Create src/app/api/quotes/[token]/approve/route.ts** — Route Handler:
- POST endpoint (no auth — customer-facing).
- Extracts token from params. Verifies via `verifyQuoteToken(token)`. If invalid → 410 response.
- Parses request body: `{ action, signatureName?, declineReason?, changeNote?, selectedOptionalItemIds? }`.
- All DB operations use `adminDb` (no RLS — customer has no auth).

- **action = 'approve':**
  - Update quote: status='approved', approved_at=now(), signature_name, approved_optional_item_ids.
  - **Auto-convert WO:** Update the parent work_order status to 'approved'. This is the WORK-05 requirement: "Approved quotes auto-convert to work orders." Call the equivalent of updateWorkOrderStatus logic but using adminDb directly (since this is an unauthenticated endpoint):
    - Set work_orders.status = 'approved'.
    - Append activity_log event: `{ type: 'quote_approved', at: now, by_id: null, note: 'Customer approved quote #{number}' }`.
  - If optional items were excluded by customer, update the WO line items: set excluded items to is_optional=true and mark them as "customer excluded" (or remove them from the WO). The approach: keep all items on the WO but store which optional items were approved in quote.approved_optional_item_ids. The invoice preparation (Plan 07) uses this list to include only approved items.
  - Notify office: insert an alert (type: 'quote_approved', message: 'Customer approved quote #{number} for {woTitle}', reference_id: wo_id).

- **action = 'decline':**
  - Update quote: status='declined', declined_at=now(), decline_reason.
  - Notify office: insert alert (type: 'quote_declined', message: 'Customer declined quote #{number}: {reason}').
  - WO status stays as 'quoted' — office can revise and re-send.

- **action = 'request_changes':**
  - Update quote: status='changes_requested', change_note.
  - Notify office: insert alert (type: 'quote_changes_requested', message: 'Customer requested changes on quote #{number}').
  - WO status stays as 'quoted'.

- Return `{ success: true }` on success, appropriate error responses on failure.
