**2. Create PhotoCapture component** in `src/components/field/photo-capture.tsx`:

Props: `{ visitId: string, orgId: string }`

**Camera button:**
- Per locked decision: "Quick camera button: tap to snap, photo auto-attaches to the visit"
- Large camera icon button (44px+) at the top
- Uses `<input type="file" accept="image/*" capture="environment" />` — triggers device camera on mobile
- Hidden input, triggered by the visible button tap

**Photo flow on capture:**
1. File selected from camera
2. Compress immediately using `browser-image-compression`:
   - `maxSizeMB: 0.3` (target ~300KB)
   - `maxWidthOrHeight: 1200` pixels
   - `useWebWorker: true` (non-blocking compression)
   - `fileType: "image/webp"` (smaller than JPEG)
3. CRITICAL (per Research pitfall 3): Write compressed blob to `offlineDb.photoQueue` IMMEDIATELY, before attempting upload. This ensures the photo is never lost if the app closes.
4. If online: request signed upload URL, upload directly to Supabase Storage, update photoQueue item status to "uploaded" and set storagePath
5. If offline: blob stays in photoQueue with status "pending" — uploaded later by photo sync processor

**Photo tag selector:**
- Per locked decision: "After capture, tech can optionally tag the photo (before / after / issue / equipment) — tagging is skippable"
- After photo is captured, show a brief tag selector overlay (small pill buttons: Before | After | Issue | Equipment | Skip)
- Skip is the default action (auto-dismisses after 3 seconds if no tag selected)
- Selected tag is stored in the photoQueue item

**Photo grid:**
- Display captured photos in a 3-column grid below the camera button
- Each photo shows thumbnail + tag badge (if tagged)
- Tap photo to view full-size in a modal/sheet
- Per locked decision: "Soft limit: no hard cap on photos, but warning shown after 10 photos per visit"
  - After 10 photos: show amber warning bar "10+ photos may slow upload. Continue?"
  - Do NOT prevent adding more

**Photo sync processor** (add to sync.ts or as a separate function called from initSyncListener):
- On connectivity return: iterate `offlineDb.photoQueue.where("status").equals("pending")`
- For each: get signed URL → upload → mark "uploaded" → set storagePath
- On failure: mark "failed", log error, retry on next connectivity event
- IMPORTANT: This is separate from the text `enqueueWrite` queue because blobs cannot be JSON-serialized

- Allowed MIME types: image/webp, image/jpeg, image/png
- RLS policy on `storage.objects`: org-scoped access using `storage.foldername(name)[1]` = org_id from JWT
