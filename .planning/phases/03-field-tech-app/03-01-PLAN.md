**2. Create `route_days` table** in `route-days.ts`:
- `id` uuid PK defaultRandom
- `org_id` uuid FK to orgs (cascade)
- `tech_id` uuid FK to profiles (set null)
- `date` date not null — the service date
- `stop_order` jsonb not null — ordered array of `{ customer_id, pool_id, sort_index }` objects
- `created_at` timestamp default now
- Unique constraint on `(org_id, tech_id, date)` — one route per tech per day
- Index on `(tech_id, date)` for fast "today's route" lookups
- RLS: 4 policies matching existing pattern
  - SELECT: all org members
  - INSERT/UPDATE: owner + office only (techs cannot create routes — Phase 4)
  - DELETE: owner + office only
- NOTE: This is a minimal table Phase 4 replaces with a full scheduling system. Phase 3 uses it to show today's stops.

**3. Create `checklist_templates` table** in `checklists.ts`:
- `id` uuid PK defaultRandom
- `org_id` uuid FK to orgs (cascade)
- `name` text not null — e.g., "Weekly Maintenance"
- `service_type` text — "routine" | "opening" | "closing" | "green_pool" | null
- `created_at` timestamp default now
- RLS: SELECT for all org members; INSERT/UPDATE/DELETE for owner+office

**4. Create `checklist_tasks` table** in same `checklists.ts` file:
- `id` uuid PK defaultRandom
- `org_id` uuid FK to orgs (cascade) — needed for RLS
- `template_id` uuid FK to checklist_templates (cascade, nullable)
- `customer_id` uuid FK to customers (cascade, nullable) — null = template-level, non-null = customer override
- `label` text not null — "Skim surface"
- `is_required` boolean default true
- `sort_order` integer default 0
- `is_deleted` boolean default false — soft delete for customer override removal
- RLS: SELECT for all org members; INSERT/UPDATE/DELETE for owner+office

**5. Create `visit_photos` table** in `visit-photos.ts`:
- `id` uuid PK defaultRandom
- `visit_id` uuid FK to service_visits (cascade)
- `org_id` uuid FK to orgs (cascade)
- `storage_path` text not null — e.g., "{org_id}/visits/{visit_id}/photo.webp"
- `tag` text nullable — "before" | "after" | "issue" | "equipment"
- `created_at` timestamp default now
- RLS: SELECT for all org members; INSERT for owner+office+tech; UPDATE/DELETE for owner+office
- Note: tech can INSERT photos for their own visits but cannot delete

- KEEP existing `version(1).stores({...})` unchanged — never modify existing versions (see Research pitfall 2)
- Add `version(2).stores({...})` with ALL stores (existing + new):
  - `syncQueue`: same as v1 (`"++id, createdAt, retries, status"`)
  - `routeCache`: same as v1 (`"id, cachedAt, expiresAt"`)
  - `visitDrafts`: `"id, stopId, updatedAt, status"` — id is visit UUID (string, not auto-increment)
  - `photoQueue`: `"++id, visitId, status, createdAt"` — auto-increment id; blob column NOT indexed (critical — indexing blobs corrupts IDB performance)

**3. Add TypeScript interfaces:**

```typescript
export interface VisitDraft {
  id: string                    // visit UUID generated client-side
  stopId: string                // route stop identifier
  poolId: string                // pool being serviced
  customerId: string            // customer ID
  chemistry: Record<string, number | null>  // all reading values keyed by param name
  checklist: Array<{ taskId: string; completed: boolean; notes: string }>
  notes: string
  status: "draft" | "completed"
  updatedAt: number             // Date.now()
}

export interface PhotoQueueItem {
  id?: number                   // auto-increment
  visitId: string               // FK to visitDrafts.id
  blob: Blob                    // raw compressed image — NOT base64, NOT indexed
  tag?: "before" | "after" | "issue" | "equipment"
  status: "pending" | "uploaded" | "failed"
  storagePath?: string          // filled after successful upload
  createdAt: number
}
```

**4. Add typed Table properties to the OfflineDB class:**
```typescript
visitDrafts!: Table<VisitDraft>
photoQueue!: Table<PhotoQueueItem>
```

IMPORTANT: Do NOT modify the version(1) block. Dexie auto-migrates from v1 to v2 when it detects the version difference.
