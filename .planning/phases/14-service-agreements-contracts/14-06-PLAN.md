  <action>
Add lifecycle server actions to `src/actions/agreements.ts`:

**`pauseAgreement(id, reason?)`:**
1. Verify agreement status is 'active'
2. Update agreement: status = 'paused', paused_at = now, paused_reason = reason
3. Deactivate all linked schedule rules: for each pool_entry with a schedule_rule_id, set `schedule_rules.active = false`
4. Append to activity_log: `{ type: "agreement_paused", at, by_id, note: "Agreement paused. Reason: {reason}" }`
5. Return updated agreement

**`resumeAgreement(id)`:**
1. Verify status is 'paused'
2. Update agreement: status = 'active', paused_at = null, paused_reason = null
3. Reactivate schedule rules: for each pool_entry with a schedule_rule_id, set `schedule_rules.active = true`. Update `anchor_date` to today's date (using `toLocalDateString()`) so the next stop generates from the resume date, not the original start.
4. Append to activity_log: `{ type: "agreement_resumed", at, by_id, note: "Agreement resumed" }`
5. Return updated agreement

**`cancelAgreement(id)`:**
1. Verify status is 'active' or 'paused'
2. Fetch org_settings for `agreement_notice_period_days`
3. If notice_period_days > 0: calculate effective cancellation date = today + notice_period_days. Set `end_date` to that date (the agreement remains active until then). Append log noting the notice period.
4. If notice_period_days = 0: immediate cancellation — status = 'cancelled', cancelled_at = now, deactivate all schedule rules.
5. Append to activity_log with cancellation details
6. Return updated agreement

**`renewAgreement(id)`:**
1. Verify status is 'expired' or 'active' (for early renewal)
2. Create a new draft agreement copying all fields from the current one:
   - New agreement_number (increment from org_settings)
   - status = 'draft'
   - start_date = current agreement's end_date (or today if expired)
   - end_date calculated from term_type
   - Copy all pool_entries as new rows (without schedule_rule_id — those get created on acceptance)
   - version = 1 (fresh agreement)
3. Append to original agreement's activity_log: "Renewed — new agreement SA-XXXX created"
4. Return the new draft agreement

**`checkExpiredAgreements()`:**
Server action (or utility function) that finds all agreements where:
- status = 'active'
- auto_renew = false
- end_date <= today (using toLocalDateString for comparison)
Transitions each to 'expired', deactivates schedule rules, appends to activity_log. This will be called by the cron in Plan 07.

All actions use withRls, return `{ success, data?, error? }`, and return the fresh agreement object for direct state updates in the UI (no router.refresh).

**Add amendment actions to `src/actions/agreements.ts`:**

**`amendAgreement(id, changes, changeSummary)`:**
1. Verify status is 'active'
2. Classify the amendment as major or minor based on what changed:
   - **Major (requires customer re-sign):** price change (any amount), term_type change, frequency change, add/remove pool
   - **Minor (notification only):** checklist task addition/removal, preferred_day change, notes change
3. Increment agreement.version
4. Create `agreement_amendments` row: agreement_id, version_number, amendment_type, change_summary, changed_by_id, status, snapshot_json (capture full current agreement state as JSON before applying changes)
5. Apply the changes to the agreement and/or pool_entries
6. If major:
   - Set amendment status = 'pending_signature'
   - Re-send agreement for customer approval (call sendAgreement internally — this generates a new token and sends email with the updated terms)
   - Append activity_log: "Major amendment (v{N}) — sent for customer re-approval"
7. If minor:
   - Set amendment status = 'signed' (auto-approved)
   - Send notification email to customer about the minor change (simple email, not the full agreement)
   - Update linked schedule rules if frequency/day changed
   - Append activity_log: "Minor amendment (v{N}) applied — {changeSummary}"
8. Return updated agreement

**`src/components/agreements/amendment-dialog.tsx`:**
Dialog component for creating an amendment. Shows:
- Current agreement summary (current values for editable fields)
- Editable fields grouped by category:
  - **Pricing changes** (per pool entry): amount inputs
  - **Frequency changes** (per pool entry): frequency selector, preferred day
  - **Pool changes**: add new pool (from customer's unassigned pools) or remove existing pool
  - **Term changes**: term_type selector
  - **Service changes**: checklist task selection per pool
- Change summary textarea (auto-populated based on what changed, editable)
- Classification indicator: shows "This will require customer re-approval" (red badge) for major changes or "This will take effect immediately" (green badge) for minor changes. Updates in real-time as user makes changes.
- Confirm button
