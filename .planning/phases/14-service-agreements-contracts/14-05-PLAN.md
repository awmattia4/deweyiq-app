   - Term: "Month-to-Month" or "6 Months: Jan 1 – Jun 30, 2026"

   For each pool entry, a card showing:
   - Pool name + type
   - Frequency (e.g. "Weekly — Mondays")
   - Pricing: formatted by model type
   - Service checklist: list of included tasks
   - Schedule rule link (if active — link to schedule page)

3. **Pricing Summary:**
   - Total monthly cost (sum of all pool entries)
   - Billing model breakdown

4. **Terms & Conditions:**
   - Collapsible sections for T&C, Cancellation Policy, Liability Waiver
   - Show first few lines, expandable to full text

5. **Internal Notes:**
   - Office-only notes section (not customer-visible)

**Right column — Actions & Timeline:**

1. **Action Buttons** (contextual by status):
   - Draft: "Send to Customer" (calls sendAgreement), "Edit" (links to edit page — reuse builder), "Download PDF", "Delete"
   - Sent: "Resend" (re-sends email), "Download PDF", "Cancel"
   - Active: "Pause", "Amend" (Plan 06), "Cancel", "Download PDF"
   - Paused: "Resume", "Cancel", "Download PDF"
   - Expired: "Renew" (creates new draft from this agreement), "Download PDF"
   - Cancelled/Declined: "Download PDF", "Create New from This" (duplicates as new draft)

   Buttons that require Plan 06 (pause, resume, cancel, amend) should call the lifecycle actions from `agreements.ts`. For now, wire only what's available (send, delete, PDF download). Plan 06 will wire the remaining lifecycle actions.

2. **Signature Info** (if signed):
   - Signed by: name
   - Signature image (if drawn — render the base64 PNG)
   - Signed at: timestamp
   - IP captured

3. **Activity Timeline:**
   - Render activity_log jsonb entries as a vertical timeline
   - Each entry: timestamp, action description
   - Most recent at top

4. **Amendments** (if any):
   - List of amendments with version number, type (major/minor), change summary, status

- "Send to Customer" → calls sendAgreement action → on success, update local state with returned data (do NOT use router.refresh)

- "Delete" → calls deleteAgreement → redirect to /agreements on success
- "Edit" → navigates to /agreements/new?edit=${id} (or a dedicated edit route) — the builder should support pre-filling from an existing agreement
