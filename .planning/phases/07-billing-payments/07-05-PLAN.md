    **Create `src/app/api/pay/[token]/setup/route.ts`** — POST handler for saving payment methods:
    - Verify pay token. If invalid, return 401.
    - Fetch invoice via adminDb to get customer_id and org_id.
    - Fetch org_settings for stripe_account_id.
    - Fetch or create Stripe Customer (same pattern as PaymentIntent route in Plan 04).
    - Create SetupIntent on the connected account:
      ```
      stripe.setupIntents.create({
        customer: stripeCustomerId,
        usage: "off_session",
        payment_method_types: ["card", "us_bank_account"],
      ```

    - Return `{ clientSecret: setupIntent.client_secret }`.

    **Update `src/app/pay/[token]/pay-client.tsx`** — add AutoPay opt-in:
    - After the main payment form, add an "Enable AutoPay" section:
      - Checkbox: "Save this payment method for future automatic payments"
      - When checked AND payment succeeds, also call `/api/pay/${token}/setup` to create a SetupIntent.
      - After SetupIntent confirmation via `stripe.confirmSetup()`, save the payment method ID.
    - Alternatively, simpler approach: use `setup_future_usage: "off_session"` on the existing PaymentIntent (already passed in Plan 04). This automatically saves the payment method without needing a separate SetupIntent flow.
    - **Preferred approach:** Add `setup_future_usage: "off_session"` to the PaymentIntent creation in the intent route ONLY when the customer checks the "Enable AutoPay" checkbox. Pass `saveMethod: true` in the POST body to `/api/pay/${token}/intent`.
    - When customer checks the box and payment succeeds, the PaymentMethod is saved to the Stripe Customer automatically.
    - After successful payment + AutoPay opt-in, call a server action to update the customer record:
      - Set `autopay_enabled = true`
      - Set `autopay_method_id` to the PaymentMethod ID from the PaymentIntent response.
    - Show confirmation: "AutoPay enabled. Future invoices will be charged automatically."

    **Update `src/actions/payments.ts`** — add AutoPay functions:
    - `enableAutoPay(customerId: string, paymentMethodId: string)`:
      - Uses adminDb (called from public payment page context, no user session).
      - Updates customers: autopay_enabled=true, autopay_method_id=paymentMethodId.
    - `chargeAutoPay(invoiceId: string)`:
      - Fetch invoice, customer, org_settings.
      - Verify customer.autopay_enabled and customer.autopay_method_id exist.
      - Verify org has stripe_account_id.
      - Calculate total in cents (including surcharge if applicable for card payments).
      - Create PaymentIntent with off_session + confirm:
        ```
        stripe.paymentIntents.create({
          amount: totalCents,
          currency: "usd",
          customer: customer.stripe_customer_id,
          payment_method: customer.autopay_method_id,
          off_session: true,
          confirm: true,
          application_fee_amount: surchargeEnabled ? surchargeAmountCents : undefined,
          metadata: { invoice_id: invoiceId, org_id: orgId },
        }, { stripeAccount: connectedAccountId })
        ```
      - Save stripe_payment_intent_id on the invoice.
      - Create a payment_records entry with status='pending'.
      - Do NOT mark invoice as paid — wait for webhook.
      - Return `{ success: true, paymentIntentId }`.
      - Handle `authentication_required` error: Some cards require 3DS. In this case, send the customer an email with the payment link so they can authenticate manually. Set payment_records status='requires_action'.
    - `disableAutoPay(customerId: string)`:
      - Uses withRls. Owner+office only.
      - Updates customers: autopay_enabled=false, autopay_method_id=null.

    **Update `src/actions/billing.ts`** — integrate AutoPay into invoice generation:
    - In `generateInvoiceForCustomer`: after creating the invoice, check if customer.autopay_enabled is true. If so, call `chargeAutoPay(invoiceId)`. Wrap in try/catch — AutoPay failure should not block invoice creation.
    - In `generateAllInvoices`: after all invoices are created, loop through AutoPay-enabled customers and charge sequentially (not Promise.all). Log successes and failures. Return { created, autoPay: { charged: number, failed: number } }.

    **Create `src/lib/emails/dunning-email.tsx`** — React Email template for dunning reminders:
    - Follow the same pattern as invoice-email.tsx (dark-themed, branded).
    - Props: companyName, customerName, invoiceNumber, totalAmount, paymentUrl, stepNumber, maxSteps, customSubject, customBody.
    - Default subject: "Payment Reminder: Invoice {invoiceNumber}"
    - Default body: "Your payment of ${totalAmount} for invoice {invoiceNumber} is overdue. Please pay at your earliest convenience."
    - If custom subject/body are provided from dunning_config.steps, use those instead.
    - CTA button: "Pay Now" linking to the payment URL.
    - Footer: "This is reminder {stepNumber} of {maxSteps}. If you have already made this payment, please disregard this notice."
    - All colors hex only (React Email + Resend compatibility).

    **Create `src/actions/dunning.ts`** — "use server" module:

    1. `runDunningScan(orgId?: string)`:
       - If orgId provided, scan one org. Otherwise, scan all orgs with active dunning configs.
       - For each org:
         - Load dunning_config (steps array, max_retries).
         - Query all invoices with status='overdue' or status='sent' with due_date past.
         - For each overdue invoice:
           - Find the latest payment_records entry. Count attempts.
           - If attempt_count >= max_retries, skip (max retries reached).
           - Determine which dunning step applies based on days since due_date:
             - Step 1 at day_offset 3 (default), Step 2 at day_offset 7, Step 3 at day_offset 14, etc.
           - If today matches a step's day_offset (with 1-day window to handle cron timing), AND no dunning email has been sent for this step:
             - Send dunning email via Resend using DunningEmail template.
             - If customer has autopay_enabled and a saved payment method, attempt retry via `retryPayment(invoiceId)`.
             - If customer does NOT have autopay, just send the email with pay link.
           - Track dunning progress: store last_dunning_step and last_dunning_at on the invoice (add these columns if not present — or use a simple approach: store in payment_records.metadata JSONB).
       - Use adminDb for all queries (cron job has no user session).

    2. `retryPayment(invoiceId: string)`:
       - Fetch invoice and customer.
       - If customer has autopay_method_id, create a new off-session PaymentIntent (same as chargeAutoPay in payments.ts).
       - Create a new payment_records entry with incremented attempt_count.
       - Do NOT mark invoice paid — webhook handles that.

    3. `getDunningConfig(orgId: string)`:
       - Fetch dunning_config for the org. Return with defaults if none exists.

    4. `updateDunningConfig(steps: DunningStep[], maxRetries: number)`:
       - Uses withRls. Owner only.
       - Upsert dunning_config for the org.
       - Validate: steps must have day_offset > 0, max_retries 1-10, steps.length <= 5.
       - Revalidate settings path.

    **Create `src/components/settings/dunning-settings.tsx`** — owner-configurable dunning UI:

    - Max retries: numeric input (1-10).

      - Call `runDunningScan()` (no orgId = scan all orgs).
      - Return 200 with results.
