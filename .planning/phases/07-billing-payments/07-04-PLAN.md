  <action>
    **Create `src/app/api/webhooks/stripe/route.ts`** — webhook endpoint:
    - Uses `req.text()` (NOT req.json()) to get raw body for signature verification.
    - Get `stripe-signature` header.
    - For Connect events (payment events on connected accounts), use `STRIPE_CONNECT_WEBHOOK_SECRET`.
    - For platform events (account.updated), use `STRIPE_WEBHOOK_SECRET`.
    - Determine which secret to use: Check if `event.account` exists — if yes, it's a Connect event.
    - Actually, per Stripe docs, configure two separate webhook endpoints. However, for simplicity in App Router, use a single endpoint and try both secrets:
      1. Try `stripe.webhooks.constructEvent(body, sig, STRIPE_CONNECT_WEBHOOK_SECRET)`.
      2. If that fails, try `stripe.webhooks.constructEvent(body, sig, STRIPE_WEBHOOK_SECRET)`.
      3. If both fail, return 400.
    - Route to handlers based on event.type:
      - `payment_intent.succeeded` -> `handlePaymentSucceeded`
      - `payment_intent.payment_failed` -> `handlePaymentFailed`
      - `account.updated` -> `handleAccountUpdated`
      - `charge.refunded` -> `handleChargeRefunded`
    - Return 200 after handler completes.
    - CRITICAL: This endpoint must NOT use any authentication middleware (no getCurrentUser). Stripe calls it directly.

    **Create `src/lib/stripe/webhook-handlers.ts`** — event handler functions:

    1. `handlePaymentSucceeded(paymentIntent: Stripe.PaymentIntent)`:
       - Extract `invoice_id` from `paymentIntent.metadata`.
       - Find the payment_records entry by stripe_payment_intent_id.
       - Update payment_records: status='settled', settled_at=now.
       - Update invoice: status='paid', paid_at=now, payment_method='card' or 'ach' (detect from `paymentIntent.payment_method_types` used).
       - Receipt email is NOT sent here — Plan 05 creates `src/lib/emails/receipt-email.tsx` and wires it into this handler for ALL successful payments (both manual and AutoPay). Leave a clearly marked integration point: `// TODO(Plan-05): Send receipt email via ReceiptEmail template`.
       - Use adminDb for all updates (webhook has no user session).

    2. `handlePaymentFailed(paymentIntent: Stripe.PaymentIntent)`:
       - Extract `invoice_id` from metadata.
       - Update payment_records: status='failed', failure_reason=paymentIntent.last_payment_error?.message, attempt_count increment.
       - Update invoice: status to 'overdue' if it was 'sent'.
       - Update customer.overdue_balance (add the invoice amount).
       - Create an alert in the alerts table (type='payment_failed', reference_id=invoice_id) — surfaces on alerts dashboard.
       - Use adminDb.

    3. `handleAccountUpdated(account: Stripe.Account)`:
       - Find org by stripe_account_id matching account.id.
       - Update org_settings: stripe_onboarding_done = account.charges_enabled && account.details_submitted.
       - Use adminDb.

    4. `handleChargeRefunded(charge: Stripe.Charge)`:
       - Find invoice via charge.payment_intent metadata.
       - Create a payment_records entry with method matching original, status='refunded', amount negative.
       - Update invoice status: if fully refunded, set to 'void'; if partial, keep as 'paid' but note the refund.
       - Use adminDb.

    All handlers must be idempotent — check current state before mutating. Use `payment_records.stripe_payment_intent_id` as the idempotency key. If a payment_record with 'settled' status already exists for this PI, skip.
  </action>
