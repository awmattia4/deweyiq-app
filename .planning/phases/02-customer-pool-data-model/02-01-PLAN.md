  <action>
Create four new schema files following the exact patterns from orgs.ts and profiles.ts (same imports, same RLS policy structure, same `.enableRLS()` call).

**src/lib/db/schema/customers.ts:**
- Export `customerStatusEnum` pgEnum: `["active", "paused", "cancelled"]`
- Export `customers` pgTable with columns per RESEARCH.md Schema Design Reference:
  - id (uuid PK defaultRandom), org_id (uuid NOT NULL FK -> orgs.id CASCADE), full_name (text NOT NULL), address (text), phone (text), email (text), gate_code (text), access_notes (text), status (customerStatusEnum NOT NULL default "active"), assigned_tech_id (uuid FK -> profiles.id SET NULL), route_name (text), created_at (timestamptz NOT NULL defaultNow), updated_at (timestamptz NOT NULL defaultNow)
- Indexes: org_id, status, assigned_tech_id
- RLS policies (4): select for all org members, insert/update/delete for owner+office only. Use `(select auth.jwt() ->> 'org_id')::uuid` subquery pattern (not per-row re-evaluation).
- Export `customersRelations` using v1 `relations()` API: org (one -> orgs), assignedTech (one -> profiles), pools (many -> pools)

**src/lib/db/schema/pools.ts:**
- Export `poolTypeEnum` pgEnum: `["pool", "spa", "fountain"]`
- Export `poolSurfaceEnum` pgEnum: `["plaster", "pebble", "fiberglass", "vinyl", "tile"]`
- Export `sanitizerTypeEnum` pgEnum: `["chlorine", "salt", "bromine", "biguanide"]`
- Export `pools` pgTable:
  - id (uuid PK), org_id (uuid NOT NULL FK -> orgs.id CASCADE), customer_id (uuid NOT NULL FK -> customers.id CASCADE), name (text NOT NULL), type (poolTypeEnum NOT NULL default "pool"), volume_gallons (integer nullable), surface_type (poolSurfaceEnum nullable), sanitizer_type (sanitizerTypeEnum nullable), notes (text), created_at, updated_at
- Indexes: org_id, customer_id
- RLS: same pattern as customers (select for org, insert/update/delete for owner+office)
- Export `poolsRelations`: org (one), customer (one -> customers), equipment (many), serviceVisits (many)

**src/lib/db/schema/equipment.ts:**
- Export `equipment` pgTable:
  - id (uuid PK), org_id (uuid NOT NULL FK -> orgs.id CASCADE), pool_id (uuid NOT NULL FK -> pools.id CASCADE), type (text NOT NULL — NOT a pgEnum, open-ended per research), brand (text), model (text), install_date (date nullable — use `date("install_date")`), notes (text), created_at, updated_at
- Indexes: org_id, pool_id
- RLS: same org_id gate, owner+office write
- Export `equipmentRelations`: org (one), pool (one -> pools)

**src/lib/db/schema/service-visits.ts:**
- Export `serviceVisits` pgTable (stub — Phase 3 populates):
  - id (uuid PK), org_id (uuid NOT NULL FK -> orgs.id CASCADE), customer_id (uuid NOT NULL FK -> customers.id CASCADE), pool_id (uuid FK -> pools.id SET NULL), tech_id (uuid FK -> profiles.id SET NULL), visit_type (text — "routine"/"repair"/"one_off"), visited_at (timestamptz NOT NULL), notes (text), created_at (timestamptz NOT NULL defaultNow)
- Indexes: org_id, customer_id, pool_id
- RLS: select for all org members, insert/update for owner+office+tech (techs write service records in Phase 3)
- Export `serviceVisitsRelations`: org (one), customer (one), pool (one), tech (one -> profiles)

**src/lib/db/schema/index.ts:**
- Add `export * from "./customers"`, `export * from "./pools"`, `export * from "./equipment"`, `export * from "./service-visits"` to the existing barrel.

CRITICAL: Every `pgEnum` and every `xyzRelations` export MUST be re-exported via the barrel. If missed, drizzle-kit won't generate the enum migration and relational queries will silently fail (see Pitfall 1 and 2 in RESEARCH.md).

After creating all files, generate and apply the migration:

npx drizzle-kit push
