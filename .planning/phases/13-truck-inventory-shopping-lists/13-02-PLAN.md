  <action>
    **1. `src/actions/shopping-lists.ts`** — Full shopping list server actions:

    - `getShoppingList(techId?: string)` — If techId provided, get tech's items + shared org items. If null, get all org items (for office view). Order by: is_urgent DESC, status ASC (needed first), created_at DESC. Use withRls. Include joins to parts_catalog and chemical_products for display info.

    - `addShoppingListItem(data: { itemName, category, quantityNeeded, unit, techId?, sourceType?, sourceWorkOrderId?, sourceProjectId?, sourceInventoryItemId?, isUrgent?, urgentReason?, notes?, catalogItemId?, chemicalProductId? })` — Insert into shopping_list_items. If source is 'low_inventory', link back to truck_inventory item.

    - `transitionShoppingListItem(itemId, newStatus, data?)` — Move item through lifecycle: needed -> ordered (set ordered_at, ordered_by_id, vendor, po_reference) -> received (set received_at, received_by_id) -> loaded (set loaded_at, loaded_by_id) -> used (set used_at, used_by_id). Validate transitions are sequential (can't go from 'needed' to 'loaded'). Update updated_at.

    - `autoGenerateFromWO(workOrderId)` — Read work order line items (from work_order_line_items table), create shopping_list_items for each part with source_type='work_order', source_work_order_id set. Skip items already on the list (dedup by catalog_item_id + source_work_order_id). Assign to the tech assigned to the WO.

    - `autoGenerateFromLowInventory(techId)` — Scan truck inventory for items below min_threshold, create shopping_list_items with source_type='low_inventory' and quantity_needed = (min_threshold - current_quantity). Skip if item already on list.

    - `autoGenerateFromScheduleForecast(techId, routeDate)` — Look at scheduled stops for the route date, check dosing history for those pools (average chemical usage from recent service_visits.dosing_amounts), create shopping_list_items with source_type='forecast'. This powers the "Predicted Needs" section.

    - `getPartsReadyStatus(workOrderId)` — Check if all WO line items have corresponding shopping_list_items with status='loaded' or 'used'. Returns { ready: boolean, total: number, loaded: number, items: [...] }.

    - `deleteShoppingListItem(itemId)` — Remove item (owner/office, or tech if they created it).

    - `flagUrgent(itemId, reason)` / `unflagUrgent(itemId)` — Toggle urgency.

    **2. `src/actions/what-to-bring.ts`** — Pre-route summary:

    - `getWhatToBring(techId, routeDate)` — Aggregates ALL requirements for the route:
      1. Get tech's scheduled stops for the date (from route_stops + route_days).
      2. Get WO parts for any work orders assigned to stops on this date.
      3. Get checklist requirements that imply consumables (optional, enhance later).
      4. Get predicted chemical needs from dosing history (average of last 4 visits per pool).
      5. Cross-reference ALL above against tech's current truck inventory.
      6. Return items categorized as: { missing: [...], low: [...], stocked: [...] }.
      Each item: { itemName, category, unit, quantityNeeded, quantityOnTruck, shortfall, source: 'wo'|'schedule'|'forecast', urgency: 'missing'|'low'|'stocked' }.
      Sort: missing first, then low, then stocked.
      CRITICAL: Use LEFT JOIN + GROUP BY for aggregation (never correlated subqueries inside withRls per MEMORY.md).
  </action>

    2. Shopping list lifecycle transitions enforce correct order
    3. `getWhatToBring` returns items sorted by urgency
    4. `autoGenerateFromWO` correctly deduplicates items
  </verify>
