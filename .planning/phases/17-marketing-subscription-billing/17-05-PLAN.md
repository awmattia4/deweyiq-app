  <files>
    src/actions/billing.ts
  </files>
  <action>
    Add to `src/actions/billing.ts`:

    1. `syncPoolCount(orgId?: string)` — reconciles stored pool_count with actual DB count
       - Counts pools in the pools table for the org
       - Updates subscriptions.pool_count via adminDb
       - Returns the actual count
       - If no orgId provided, uses the current user's org from RLS token
       - Called by billing page on load and after pool add/delete as safety net

    2. `checkPoolLimit()` — checks if current org can create a new pool
       - Fetches subscription for the org
       - Returns { allowed: boolean, currentCount, limit, tier, graceActive, graceEndsAt, needsUpgrade }
       - Logic:
         - If no subscription or status is "trialing"/"active" and count < limit: allowed = true
         - If count >= limit and grace_period_ends_at is null: set grace period (7 days), allowed = true (first overage)
         - If count >= limit and grace_period_ends_at > now: allowed = true (within grace)
         - If count >= limit and grace_period_ends_at <= now: allowed = false (grace expired)

    3. `updatePoolCount(delta: 1 | -1)` — increments or decrements pool_count
       - Uses adminDb to update subscriptions.pool_count
       - Called by pool add/delete actions
       - If incrementing past limit and no grace period set: sets grace_period_ends_at = now + 7 days
  </action>
  <verify>
    1. `npm run build` succeeds
    2. All three functions exported
    3. Grace period logic is correct
  </verify>
  <done>
    Pool count tracking and limit checking ready for integration into pool CRUD actions.
