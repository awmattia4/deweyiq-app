  <action>
    Install React Email dependencies:
    ```bash
    npm install @react-email/components @react-email/render
    ```

    Create `src/lib/emails/service-report-email.tsx`:
    - Export `ServiceReportEmail` React component accepting props: customerName, techName, companyName, serviceDate, poolName, chemistry (Record<string, number | null>), checklist (Array<{task: string, completed: boolean}> optional), reportUrl (string — the full public report link)
    - Use `@react-email/components`: Html, Head, Body, Container, Section, Text, Hr, Row, Column, Button
    - Dark-themed email design matching the app's dark-first design system (deep navy background #0f172a, slate text colors — per Phase 1 decision)
    - CRITICAL: Use hex colors only in email inline styles — NOT oklch() (email clients don't support oklch)
    - Email body layout:
      1. Company name header + service date
      2. "Serviced by {techName}" subheader
      3. Pool name
      4. Chemistry summary: show top 4-6 non-null readings in a 2-column grid (param name | value)
      5. Checklist summary: "X of Y tasks completed" (one line, not full checklist)
      6. Horizontal rule separator
      7. Blue CTA button: "View Full Report" linking to reportUrl
      8. Footer: "This report was generated automatically by {companyName}"
    - Export as named export (not default export)

    Create `src/lib/emails/pre-arrival-email.tsx`:
    - Simple fallback email for customers without phone numbers
    - Props: customerName, techName, stopNumber
    - Simple text email: "Hi {customerName}, your pool tech {techName} is heading your way. You're stop #{stopNumber} on today's route."
    - Same dark-themed styling as service report email
    - Keep it minimal — this is a text notification, not a marketing email

    Create `src/lib/reports/report-token.ts`:
    - Use `jose` (already bundled with Next.js — no install needed)
    - `signReportToken(visitId: string): Promise<string>` — signs a JWT with `{ visitId }` payload, HS256 algorithm, 30-day expiry. Secret from `process.env.REPORT_TOKEN_SECRET`
    - `verifyReportToken(token: string): Promise<{ visitId: string } | null>` — verifies and returns payload, or null on any error (expired, invalid signature, malformed)
    - REPORT_TOKEN_SECRET must be a 32+ character random string set in .env.local

    Create `src/app/api/reports/[token]/route.ts`:
    - Next.js Route Handler with `GET` export
    - Verify the token using `verifyReportToken(params.token)`
    - If invalid/expired: return Response("Link expired or invalid", { status: 410 })
    - If valid: query `service_visits.report_html` using `adminDb` (service role — this is a public endpoint, no Supabase auth)
    - If no report found: return Response("Report not found", { status: 404 })
    - Return the HTML string with Content-Type: text/html
    - This route requires NO authentication — the signed token IS the authorization
  </action>
  <verify>
    1. `npm run build` succeeds — React Email components compile
    2. report-token.ts exports signReportToken and verifyReportToken
    3. API route file exists at src/app/api/reports/[token]/route.ts
    4. No oklch() colors in email template inline styles
  </verify>
  <done>
    React Email templates ready for rendering. Report token system signs/verifies JWTs for public links. API route serves report HTML from signed tokens without requiring login.
  </done>

<task type="auto">
  <name>Task 2: Wire React Email report into completeStop action flow</name>
  <files>
    src/actions/visits.ts

  <action>
    Update `src/actions/visits.ts` completeStop function to use the new React Email template:

    1. Import `renderAsync` from `@react-email/render` and `ServiceReportEmail` from `@/lib/emails/service-report-email`
    2. Import `signReportToken` from `@/lib/reports/report-token`
    3. In the completeStop flow, AFTER the service visit is saved to DB:
       a. Generate a signed report token: `const reportToken = await signReportToken(visitId)`
       b. Build the report URL: `const reportUrl = ${process.env.NEXT_PUBLIC_APP_URL}/api/reports/${reportToken}`
       c. Render the React Email template to HTML: `const reportHtml = await renderAsync(ServiceReportEmail({ ... }))`
       d. Save the rendered HTML to `service_visits.report_html` (this replaces the existing `generateServiceReport()` call)
       e. If customer has email AND this is NOT an update (check the existing `isUpdate` guard):
          - Invoke `send-service-report` Edge Function with the new `reportHtml`
       f. If customer has NO email: skip silently — no error, no alert (per locked decision)

    4. On stop EDITS (isUpdate = true):
       - Still update report_html in the DB (so the public link serves fresh content)
       - Do NOT re-invoke send-service-report (email already sent — per locked decision)
       - Do NOT generate a new token (old token still works — it references visitId, not HTML)

    5. Remove or deprecate the call to the old `generateServiceReport()` function from `src/lib/reports/service-report.ts`. Do NOT delete the file yet — just remove the import and call in completeStop. The old file can be cleaned up later.

    CRITICAL: The existing `send-service-report` Edge Function does not change — it already accepts `reportHtml` in the body and sends via Resend. The only change is WHAT HTML it receives (React Email output instead of raw string builder output).

    CRITICAL: There is a non-blocking prettier version warning when @react-email/render loads with Turbopack. This is a known issue (react-email discussions/1816). Do NOT add serverExternalPackages unless compilation actually fails.
  </action>
  <verify>
    1. `npm run build` succeeds
    2. completeStop no longer calls generateServiceReport() from the old module
    3. completeStop calls renderAsync(ServiceReportEmail(...))
    4. completeStop generates and stores reportToken-based URL
    5. isUpdate guard prevents re-sending email on edits
  </verify>
