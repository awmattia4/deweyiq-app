  <name>Task 1: Build alert generation logic and server actions</name>
  <files>

  <action>
    Create `src/actions/alerts.ts` with the following server actions:

    **1. generateAlerts(orgId: string)** — Detects and inserts new alerts:

    a) **Missed stops:** Query route_stops where scheduled_date < today AND status NOT IN ("complete", "skipped", "holiday"). For each, check if an alert with same (org_id, "missed_stop", reference_id=stop.id) already exists and is not dismissed. If not, insert a new alert row.
       - Title: "{customerName}'s stop was missed on {scheduled_date}"
       - Severity: "critical" (missed service is high priority)
       - reference_type: "route_stop"
       - metadata: { customerId, poolId, techId, scheduledDate }

    b) **Incomplete data:** Query service_visits where status = "complete" AND completed_at is within the last 7 days. For each, check if chemistry_readings is null or has fewer than 2 non-null values. If incomplete, check for existing alert. If not exists, insert.
       - Title: "{customerName}'s service on {date} has incomplete data"
       - Severity: "warning"
       - reference_type: "service_visit"
       - metadata: { customerId, poolId, techId, missingParams: [...] }
       - Note: The "required params" check will be enhanced in Plan 05 when org_settings.required_chemistry_by_sanitizer is wired in. For now, use a baseline check (at least 2 chemistry readings present).

    c) **Declining chemistry:** For each pool with 3+ visits in the last 30 days, compute the slope of key chemistry parameters (freeChlorine, pH, totalAlkalinity) across the last 3 readings.
       - Use simple slope: `(last_value - first_value) / (n - 1)`
       - Decline thresholds (Claude's discretion): freeChlorine: -0.5 ppm/visit, pH: -0.15 units/visit, totalAlkalinity: -5 ppm/visit, salt: -100 ppm/visit
       - If any parameter's slope exceeds its decline threshold, insert alert.
       - Title: "{param} is declining for {customerName}'s {poolName}"
       - Severity: "warning"
       - reference_type: "service_visit" (reference the most recent visit)
       - metadata: { customerId, poolId, param, slope, values: [...] }

    Use `ON CONFLICT DO NOTHING` on the unique constraint (org_id, alert_type, reference_id) to prevent duplicates. This is the primary deduplication mechanism.

    Use `adminDb` for alert generation (not withRls) because the generation process needs to scan across all org data without RLS filtering complexities. The caller already validates the user's org membership.

    **2. getActiveAlerts()** — Returns active alerts for the current user's org:
    - Use withRls to query alerts where dismissed_at IS NULL AND (snoozed_until IS NULL OR snoozed_until < now())
    - Order by: severity DESC (critical > warning > info), then generated_at DESC (newest first)
    - Map severity to sort order: critical=3, warning=2, info=1

    **3. getAlertCount()** — Returns count of active alerts (for sidebar badge):
    - Same filter as getActiveAlerts but returns count only
    - This will be called from the sidebar server component

    **4. dismissAlert(alertId: string)** — Sets dismissed_at to now():
    - Use withRls to update
    - revalidatePath("/alerts")
    - revalidatePath("/dashboard")

    **5. snoozeAlert(alertId: string, durationMs: number)** — Sets snoozed_until:
    - Calculate snoozed_until = new Date(Date.now() + durationMs)
    - Use withRls to update
    - revalidatePath("/alerts")

    Export SNOOZE_OPTIONS constant:
    ```typescript
    export const SNOOZE_OPTIONS = [
      { label: "1 hour", ms: 1 * 60 * 60 * 1000 },
      { label: "4 hours", ms: 4 * 60 * 60 * 1000 },
      { label: "1 day", ms: 24 * 60 * 60 * 1000 },
      { label: "1 week", ms: 7 * 24 * 60 * 60 * 1000 },
    ] as const
    ```

    CRITICAL: Use LEFT JOIN pattern for any query that joins alerts with other RLS-protected tables. Do NOT use correlated subqueries inside withRls transactions (per MEMORY.md).
  </action>

    3. generateAlerts uses ON CONFLICT DO NOTHING for deduplication
    4. getActiveAlerts filters snoozed alerts that have expired (snoozed_until < now())
    5. No correlated subqueries inside withRls
  </verify>
