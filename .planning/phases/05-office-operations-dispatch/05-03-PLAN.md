  <action>
    Create `src/actions/notifications.ts` with a `sendPreArrivalNotifications` server action:

    1. Accept parameters: `techId: string` (the tech starting their route)
    2. Get RLS token via getClaims/getRlsToken (existing pattern)
    3. Query today's route stops for this tech using withRls:
       - Join route_stops with customers to get: customer full_name, phone, email, notifications_enabled
       - Filter: route_stops.tech_id = techId, route_stops.scheduled_date = today (YYYY-MM-DD format), route_stops.status IN ("scheduled", "in_progress"), route_stops.pre_arrival_sent_at IS NULL (idempotency check)
       - Order by sort_index ASC
       - CRITICAL: Use LEFT JOIN pattern (not correlated subquery) per MEMORY.md RLS pitfall
    4. Get tech profile name via withRls query on profiles table
    5. Filter stops: only include where notifications_enabled = true AND (phone IS NOT NULL OR email IS NOT NULL)
    6. If no eligible stops, return early with { sent: 0 }
    7. Build the stops payload for the Edge Function:
       ```typescript
       const stopsPayload = eligibleStops.map((stop, idx) => ({
         stopId: stop.routeStopId,
         customerName: stop.customerName,
         customerPhone: stop.phone,
         customerEmail: stop.email,
         stopNumber: stop.sortIndex,  // use actual sort_index, not array index
         notificationsEnabled: true,  // already filtered
       }))
       ```
    8. Invoke the `send-pre-arrival` Edge Function via Supabase client:
       ```typescript
       await supabase.functions.invoke("send-pre-arrival", {
         body: { orgId, techName, stops: stopsPayload }
       })
       ```
    9. Return { sent: stopsPayload.length }

    Also add a `startRoute` action (or extend existing route actions in routes.ts if more appropriate) that:
    1. Calls sendPreArrivalNotifications(techId)
    2. Returns the result to the UI
    3. This is the action that the tech triggers when starting their route

    Use "use server" directive. Follow existing server action patterns in the codebase (getClaims, withRls, error handling with try/catch).
  </action>

    4. Function checks pre_arrival_sent_at IS NULL for idempotency
    5. Function filters by notifications_enabled = true
  </verify>
