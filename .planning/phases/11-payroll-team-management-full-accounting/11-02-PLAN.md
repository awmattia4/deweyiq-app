<task type="auto">
  <name>Task 1: Create time tracking server actions</name>
  <files>src/actions/time-tracking.ts</files>
  <action>
    Create `src/actions/time-tracking.ts` with these server actions:

    1. **`getActiveShift()`** — Returns the current open time_entry for the authenticated tech (clocked_in_at NOT NULL, clocked_out_at IS NULL). Returns null if no active shift. Also returns break state (is there an open break_event with ended_at IS NULL?). Used by ClockInBanner to determine display state.

    2. **`getTimeTrackingEnabled()`** — Checks org_settings.time_tracking_enabled for the current user's org. Returns boolean. Used to conditionally show/hide the entire clock-in UI.

    3. **`clockIn(lat?: number, lng?: number)`** — Creates a new time_entry row. Validates: time tracking enabled, user is tech or owner, no existing open shift. Uses `toLocalDateString()` for work_date (CRITICAL: not toISOString per MEMORY.md). Captures GPS coordinates if provided. Returns { success, entryId, error }.

    4. **`clockOut(lat?: number, lng?: number)`** — Finds the active shift, sets clocked_out_at, clock_out_lat/lng, calculates total_minutes and break_minutes (sum of completed break_events durations), sets status='complete'. Returns { success, totalMinutes, breakMinutes, error }. Fire-and-forget pattern: after successful clock-out, trigger `pushTimeEntryToQbo()` in the background (implementation in Plan 04; for now, add a TODO comment).

    5. **`startBreak()`** — Creates a break_event row linked to active shift. Validates: active shift exists, no open break already. Updates time_entry status to 'on_break'. Returns { success, breakId, error }.

    6. **`endBreak()`** — Finds the open break_event (ended_at IS NULL), sets ended_at. Updates time_entry status back to 'active'. Returns { success, durationMinutes, error }.

    7. **`checkBreakCompliance(timeEntryId: string)`** — Checks if a tech has been clocked in for > overtime threshold (from org_settings.break_auto_detect_minutes, default 4 hours) without any break. If so, creates an alert via adminDb (alert type: 'break_compliance'). Called by auto-break detection in Plan 03. For now, export as a standalone function.

    All actions use `getRlsToken()` + role checks. Tech can only manage own shifts. Owner/office can read all. Use `withRls` for insert/update. Use `adminDb` only where RLS doesn't apply (alert creation, org_settings check).

    Revalidate `/routes` after clock-in/out and break start/end.
  </action>
  <verify>
    `npm run build` passes — all actions compile with correct types
  </verify>

    7 server actions exist for clock-in/out lifecycle, break management, and compliance checking. All use proper auth, RLS, and the local date helper.
  </done>
