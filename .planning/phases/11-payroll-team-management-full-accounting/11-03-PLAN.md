    **New file: `src/lib/geo/geofence.ts`** — Pure TypeScript module (no Next.js/Supabase imports):
    1. `haversineDistance(lat1, lng1, lat2, lng2)` — Returns distance in meters between two coordinates using Haversine formula. Earth radius = 6,371,000 meters.
    2. `isInsideGeofence(userLat, userLng, stopLat, stopLng, radiusMeters)` — Returns boolean. Uses haversineDistance.
    3. Export types: `GeofenceState = { stopId: string; inside: boolean; arrivedAt: number | null; enteredAt: number | null }` — tracks per-stop geofence state with debounce timing.

    CRITICAL anti-bounce pattern per research Pitfall 5: Do NOT record arrival on first GPS update inside geofence. Require tech to be inside for >= 30 seconds before recording arrival. Only record departure after being outside for >= 60 seconds. State machine per stop: `outside -> entering (30s timer) -> inside -> leaving (60s timer) -> outside`.

    **Extend `src/hooks/use-gps-broadcast.ts`**:
    - Add geofence detection to the existing GPS watch callback. On each position update:
      1. Get today's stops with their lat/lng (pass as a prop or parameter — stops are already available on the routes page).
      2. Get the org's geofence radius from org_settings (pass as prop).
      3. For each stop, check `isInsideGeofence(currentLat, currentLng, stopLat, stopLng, radius)`.
      4. Maintain a `Map<stopId, GeofenceState>` ref tracking per-stop state.
      5. On confirmed arrival (inside for >= 30s): call `recordStopArrival(stopId)` server action.
      6. On confirmed departure (outside for >= 60s after being inside): call `recordStopDeparture(stopId)` server action.
    - Only perform geofence checks if there is an active time_entry (tech is clocked in). If not clocked in, skip geofence detection but still broadcast GPS for dispatch map.
    - The existing GPS broadcast for dispatch map MUST continue working unchanged. Geofence detection is additive.
    - Accept new optional props: `stops?: { id: string; lat: number; lng: number }[]`, `geofenceRadius?: number`, `activeShiftId?: string | null`.

<task type="auto">
  <name>Task 2: Create stop arrival/departure server actions with drive time calculation</name>
  <files>src/actions/time-tracking.ts</files>
  <action>
    Add to `src/actions/time-tracking.ts` (extending from Plan 02):

    1. **`recordStopArrival(routeStopId: string, timeEntryId?: string)`** — Creates or updates a `time_entry_stops` row with `arrived_at = now()`. If a previous stop's `departed_at` exists for the same shift, calculates `drive_minutes_to_stop` as the difference between previous stop's departed_at and this stop's arrived_at. Validates: tech is clocked in (active shift exists). Uses withRls. Idempotent: if arrival already recorded for this stop+shift, skip.

    2. **`recordStopDeparture(routeStopId: string)`** — Updates the `time_entry_stops` row: sets `departed_at = now()`, calculates `onsite_minutes` as difference between arrived_at and departed_at. Validates: arrival was recorded. Uses withRls. Idempotent: if departure already recorded, skip.

    3. **`getStopTimingForShift(timeEntryId: string)`** — Returns all time_entry_stops for a given shift, ordered by arrived_at. Used by timesheets (Plan 04) to show per-stop breakdown. Returns array of { routeStopId, arrivedAt, departedAt, onsiteMinutes, driveMinutesToStop }.

    4. **`autoDetectBreak(timeEntryId: string)`** — Called from a periodic check (client-side timer or edge function cron). Checks if the tech has been clocked in with no GPS movement for > org_settings.break_auto_detect_minutes. If so, creates a break_event with `is_auto_detected = true`, started_at = last known active timestamp. Per CONTEXT: "Break handling: manual 'Take Break' button AND auto-detection of idle gaps as safety net — both modes active simultaneously."

    Drive time calculation detail: When recording arrival at stop N, look up the most recent departed_at from stop N-1 in the same shift (ordered by departed_at DESC, limit 1). The drive time = arrived_at(N) - departed_at(N-1) in minutes.
  </action>
