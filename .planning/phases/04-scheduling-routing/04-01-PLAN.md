    **1. Create `src/lib/db/schema/route-stops.ts`:**
    Define `routeStops` pgTable matching the RESEARCH.md schema pattern:
    - id (uuid, defaultRandom, primaryKey)
    - org_id (uuid, notNull, references orgs.id, onDelete cascade)
    - tech_id (uuid, nullable, references profiles.id, onDelete set null)
    - customer_id (uuid, notNull, references customers.id, onDelete cascade)
    - pool_id (uuid, nullable, references pools.id, onDelete set null)
    - schedule_rule_id (uuid, nullable, references scheduleRules.id, onDelete set null)
    - scheduled_date (text, notNull) -- 'YYYY-MM-DD' format matching Phase 3 date pattern
    - sort_index (integer, notNull) -- position in the day's route, 1-based
    - position_locked (boolean, notNull, default false) -- locked stops excluded from optimizer
    - window_start (time, nullable) -- time window constraint start
    - window_end (time, nullable) -- time window constraint end
    - status (text, notNull, default 'scheduled') -- scheduled | in_progress | complete | skipped | holiday
    - created_at, updated_at (timestamp with timezone, defaultNow)

    Indexes:
    - route_stops_org_date_idx on (org_id, scheduled_date)
    - route_stops_tech_date_idx on (tech_id, scheduled_date)
    - route_stops_schedule_rule_idx on (schedule_rule_id)
    - UNIQUE constraint on (org_id, customer_id, pool_id, scheduled_date) for upsert idempotency

    RLS policies (4 standard policies):
    - SELECT: all org members (org_id matches JWT org_id)
    - INSERT: owner + office only
    - UPDATE: owner + office + tech (tech needs to update status to in_progress/complete)
    - DELETE: owner + office only

    Use the `(select auth.jwt() ->> 'org_id')::uuid` subquery wrapping pattern from Phase 1 for performance.
    For UPDATE, tech role should only be able to update status field conceptually, but RLS is row-level not column-level, so the UPDATE policy allows tech role. The application layer (server actions) enforces which fields techs can update.

    Call `.enableRLS()` on the table.

    **1. Update `fetchStopsForTech` in `src/actions/routes.ts`:**

    The current implementation reads from `route_days.stop_order` JSONB array and joins with customers/pools. Update it to read from the new `route_stops` table instead:

    - Query `route_stops` WHERE `tech_id = techId AND scheduled_date = date AND org_id = orgId`, ordered by `sort_index` ASC
    - JOIN with customers and pools tables (same approach as current: fetch all org customers/pools into Maps, then map over stops)
    - Include `position_locked`, `window_start`, `window_end`, `status` from route_stops
    - Keep the `todayVisitStatusMap` logic for cross-referencing service_visits to get actual completion status

    Update the `RouteStop` interface to add new fields:
    - `routeStopId: string` (the route_stops.id -- replaces routeDayId as primary reference)
    - `positionLocked: boolean`
    - `windowStart: string | null`
    - `windowEnd: string | null`
    - `scheduleRuleId: string | null`
    - Keep `routeDayId` as optional/deprecated for backward compat during transition

    **Backward compatibility:** During the transition period, if no route_stops rows exist for the tech+date, fall back to reading route_days.stop_order (the Phase 3 path). This ensures the tech app works before office staff have migrated all routes to the new system. Log a warning when falling back.

    **2. Update `reorderStops` in `src/actions/routes.ts`:**

    Replace the route_days JSONB update with individual route_stops `sort_index` updates:
    - Accept `routeStopId` and `newSortIndex` pairs (or an ordered array of routeStopIds)
    - Update each route_stop's `sort_index` to its new position
    - Tech role still gets client-only reorder (Dexie) since UPDATE policy now includes tech for status changes, but we want techs to persist reorder too. UPDATE RLS allows tech role on route_stops, so tech reorder now persists to server.

    **3. Update `GET /api/routes/today` in `src/app/api/routes/today/route.ts`:**

    This endpoint duplicates `fetchStopsForTech` logic. Simplify it to call `fetchStopsForTech` directly from the shared helper (it already exists in actions/routes.ts). The API route should:
    - Authenticate (same as now)
    - Determine techId (same as now)
    - Call `fetchStopsForTech(claims, orgId, techId, today)` directly
    - Return the result as JSON

    This eliminates the duplicated query code. Import `fetchStopsForTech` -- note it's not exported currently, so export it from actions/routes.ts (it's a helper, not a server action, so it can be exported for use by the API route).

    **4. Create `src/actions/schedule.ts`:**

    Create the initial schedule server actions file with:
    - `migrateRouteDaysToRouteStops()`: A one-time migration action callable by owner/office that:
      1. Reads all route_days rows for the org
      2. For each route_day, creates route_stops rows from the stop_order JSONB array
      3. Maps each JSONB entry to a route_stop row: org_id, tech_id (from route_day), customer_id, pool_id (from stop entry), scheduled_date (from route_day.date), sort_index (from stop entry), status 'scheduled'
      4. Uses upsert (onConflictDoNothing on the unique constraint) to be idempotent
      5. Returns count of migrated stops

    - `getStopsForDay(techId: string, date: string)`: Fetches route_stops for a tech+date, used by the route builder UI in subsequent plans. Returns route stops with customer/pool names joined.

    - `updateStopOrder(stops: Array<{id: string, sortIndex: number}>)`: Updates sort_index on route_stops. Owner/office only.

    - `assignStopToRoute(customerId: string, poolId: string, techId: string, date: string, sortIndex: number)`: Creates a new route_stop row. Owner/office only.

    - `removeStopFromRoute(routeStopId: string)`: Deletes a route_stop row. Owner/office only.
