  <name>Task 1: Safety check logic and cron endpoint</name>
  <files>src/actions/safety.ts, src/app/api/cron/safety-check/route.ts, src/lib/db/schema/orgs.ts</files>
  <action>
    **Org settings extension** — Add to org_settings schema (or org_settings JSONB if it's a JSONB field):
    - `safety_timeout_minutes`: integer, default 30. How long without activity before alerting.
    - `safety_escalation_chain`: jsonb, default `[{"role": "owner", "delay_minutes": 0}]`. Array of `{ role: 'owner' | 'office' | string (specific user ID), delay_minutes: number }`.

    **Safety actions** — Create `src/actions/safety.ts`:
    - `checkUnresponsiveTechs(orgId: string)`:
      1. Get all active routes today (route_stops with scheduled_date = today and at least one stop completed, indicating route has started).
      2. For each tech with an active route:
         a. Get last_activity_at: `MAX(completed_at)` from route_stops for today where tech_id matches and completed_at is not null. This is the most recent stop completion timestamp. **Do NOT reference a "GPS cache table" — no such table exists.** The safety check uses completed stop timestamps as the activity signal. If the tech hasn't completed any stop yet but has started (first stop in progress), use the route start time (the earliest `started_at` or `pre_arrival_sent_at` from today's stops).
         b. If `now() - last_activity_at > safety_timeout_minutes`, the tech is "unresponsive."
      3. For unresponsive techs, check if a safety alert already exists (active, not dismissed). If not, create one.
      4. Execute the escalation chain:
         - At delay_minutes=0: notify the first person in the chain (e.g., owner).
         - At subsequent delay intervals: notify the next person. Track which escalation step was executed in alert metadata.
      5. Use `notifyUser()` from dispatch.ts with urgency='needs_action' and type='safety_alert'.
      6. Also send SMS to the escalation contact (not just in-app — safety is critical).

    - `dismissSafetyAlert(alertId: string)`: Tech dismisses the safety alert (e.g., was on lunch, phone died). Updates the alert with dismissed_at. Notifies the escalation chain that the tech is OK.

    **Cron endpoint** — Create `src/app/api/cron/safety-check/route.ts`:
    - GET handler with CRON_SECRET auth.
    - Runs every 5 minutes (configured externally).
    - Queries all org IDs, calls `checkUnresponsiveTechs` for each.
    - Returns summary: `{ checked: N, alertsCreated: N }`.
  </action>
  <verify>Run `npx tsc --noEmit` — no type errors. Run `npm run build` — compiles.</verify>
  <done>Safety cron detects unresponsive techs using completed stop timestamps. Escalation chain is executed. Techs can dismiss false positives.</done>
