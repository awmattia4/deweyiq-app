  <files>src/lib/db/schema/weather-proposals.ts, src/lib/db/schema/index.ts, src/lib/db/schema/relations.ts, src/lib/weather/reschedule-engine.ts, src/actions/weather.ts</files>
  <action>
    **Schema** — Create `src/lib/db/schema/weather-proposals.ts`:
    ```
    weather_reschedule_proposals table:
    - id: uuid PK
    - org_id: uuid FK(orgs), not null
    - affected_date: date (the day with bad weather)
    - weather_type: text ('rain' | 'storm' | 'heat' | 'wind')
    - weather_label: text (human-readable, e.g., "Heavy rain forecast")
    - forecast_data: jsonb (raw forecast snapshot for audit)
    - affected_stops: jsonb (array of { stopId, customerId, customerName, poolName, techId, techName, originalDate })
    - proposed_reschedules: jsonb (array of { stopId, newDate, newTechId, reason })
    - status: text ('pending' | 'approved' | 'denied' | 'expired') default 'pending'
    - notify_customers: boolean default true (per user decision: auto-notify with opt-out)
    - excluded_customer_ids: jsonb default '[]' (customers unchecked from notification)
    - approved_at: timestamp
    - approved_by_id: uuid FK(profiles)
    - created_at: timestamp
    ```
    RLS: owner + office SELECT/INSERT/UPDATE. Export and add to schema index.ts. Add relations in relations.ts.

    **Reschedule engine** — Create `src/lib/weather/reschedule-engine.ts`:
    - `findRescheduleSlots(orgId, affectedStops, forecastDays)`: For each affected stop, find the optimal reschedule day in the next 7-10 days by:
      1. Get weather forecast for the next 10 days (reuse Open-Meteo client).
      2. Filter to days with clear weather.
      3. For each clear day, check tech availability (count existing stops, compare to max daily capacity from org_settings or default 15).
      4. Score each (stop, day) by: geographic proximity to existing stops that day (haversine to centroid), tech's current load that day, and whether the customer has a preferred day.
      5. Return top recommendation per stop: `{ stopId, newDate, newTechId, reason }`.

    **Server actions** — Create `src/actions/weather.ts`:
    - `checkWeatherForOrg(orgId)`: Fetches forecast for the org's service area (use centroid of all customer lat/lng). For each day in next 7 days, classifies weather using `classifyWeatherDay()`. If `shouldReschedule`, queries route_stops for that date, calls `findRescheduleSlots()`, and creates a `weather_reschedule_proposals` row. Uses adminDb.
    - `approveProposal(proposalId)`: Updates proposal status to 'approved'. Executes the reschedules: for each proposed_reschedule, update the route_stop's scheduled_date and tech_id. Mark original stops as 'rescheduled_weather' status. Returns list of affected customer IDs for notification (Plan 10-08 will wire the actual notification send).
    - `denyProposal(proposalId)`: Updates status to 'denied'. No stop changes.
    - `getPendingProposals()`: Returns pending proposals for the current org.

    Run `drizzle-kit generate` to create the migration. Then verify RLS policies are correct per MEMORY.md guidance (check pg_catalog.pg_policies after push).
  </action>
