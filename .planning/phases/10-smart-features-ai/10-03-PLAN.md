<task type="auto">
  <name>Task 1: Auto-schedule engine and workload balancing server actions</name>
  <files>src/actions/schedule.ts</files>
  <action>
    Add to `src/actions/schedule.ts`:

    1. `getWorkloadBalance(weekStartDate: string)`: Queries route_stops for the given week (Mon-Fri), groups by tech_id and day, returns per-tech metrics: `{ techId, techName, totalStops, stopsPerDay: Record<string, number>, estimatedDriveMinutes }`. Uses adminDb for cross-tech visibility. Drive time estimated from ORS if coordinates available (cache per tech-day), else count-based heuristic (25 min per stop average).

    2. `autoScheduleWeek(weekStartDate: string)`: The core auto-schedule engine.
       - **Step 1**: Read all active schedule_rules for the org. Each rule defines a customer/pool, frequency (weekly, biweekly, etc.), preferred_day, and assigned_tech_id (may be null).
       - **Step 2**: Generate dates for the target week using existing `generateDatesForRule` logic. Filter to stops that should occur this week.
       - **Step 3**: For stops with assigned_tech_id and preferred_day, place them first (respect existing assignments).
       - **Step 4**: For unassigned stops, use a greedy geographic clustering algorithm:
         a. Get all techs with availability for each day.
         b. For each unassigned stop, score each (tech, day) combination by: geographic proximity to tech's existing stops that day (haversine distance to centroid), current stop count (prefer less-loaded techs), and whether the customer has a preferred day.
         c. Assign to the (tech, day) with the best score.
       - **Step 5**: Return `AutoScheduleProposal`: `{ weekStart, assignments: { stopId, techId, techName, day, customerName, poolName, isNew }[], metrics: { perTech: WorkloadMetrics[] } }`.
       - Do NOT persist to DB — this is a preview. Office approves before persisting.

    3. `applyAutoSchedule(proposal: AutoScheduleProposal)`: Persists the proposal by creating/updating route_stops for the week. Uses withRls. Revalidates /schedule.

    Note: The geographic scoring uses haversine distance (simple lat/lng math, no API call) for initial clustering. ORS is used only for the final drive-time estimate after assignment. This keeps the scoring fast.
  </action>

  <done>Auto-schedule engine generates balanced weekly route proposals. Workload balance metrics are queryable per week.</done>
