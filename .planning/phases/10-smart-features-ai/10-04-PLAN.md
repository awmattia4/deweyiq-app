  <name>Task 1: Enhance ORS VROOM with historical service durations</name>
  <files>src/actions/optimize.ts</files>
  <action>
    Enhance the existing `optimizeRoute` function in `src/actions/optimize.ts`:

    1. **Query historical service durations**: Before calling ORS VROOM, query the last 20 completed route_stops for each pool_id in the current route. For stops where both `started_at` and the corresponding service_visit `completed_at` exist, compute the median duration in seconds. Use adminDb for this cross-org query. Store as `Map<poolId, medianDurationSeconds>`.

    2. **Feed durations to VROOM**: In the VROOM jobs array, set the `service` field (in seconds) for each job:
       - If historical median exists for this pool_id, use it.
       - Otherwise, use the org-wide default: query the median of all completed stops for the org. If no data, fall back to `DEFAULT_SERVICE_DURATION_MINUTES * 60` (25 minutes = 1500 seconds per research).

    3. **Include service time in comparison**: Update `OptimizationResult` to include:
       - `currentTotalTimeMinutes` (drive + service time with current order)
       - `optimizedTotalTimeMinutes` (drive + service time with optimized order)
       - `timeSavedMinutes` now reflects total time savings including service-aware ordering (e.g., scheduling short stops near each other to create buffer time).

    4. **VROOM time windows**: If stops have `window_start`/`window_end`, convert to VROOM time_windows format (seconds from midnight). This respects customer appointment preferences during optimization.

    5. Preserve backward compatibility — if no historical data exists, the optimizer still works using default durations (same as before, but explicit).
  </action>
  <verify>Run `npx tsc --noEmit` — no type errors. Existing optimize flow still works.</verify>
