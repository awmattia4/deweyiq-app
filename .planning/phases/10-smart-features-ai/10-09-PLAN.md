    **user_notifications** — Create `src/lib/db/schema/user-notifications.ts`:
    ```
    - id: uuid PK
    - org_id: uuid FK(orgs) not null
    - recipient_id: uuid FK(profiles) not null
    - notification_type: text not null (e.g., 'stop_completed', 'wo_created', 'payment_received', 'quote_approved', 'chemistry_alert', 'weather_proposal', 'portal_message', 'service_request', 'invoice_overdue', 'route_started', 'route_finished', 'tech_assigned', 'schedule_change', 'customer_added', 'customer_cancelled', 'system_event')
    - urgency: text not null default 'informational' ('needs_action' | 'informational')
    - title: text not null
    - body: text
    - link: text (deep link, e.g., '/work-orders/[id]')
    - read_at: timestamp
    - dismissed_at: timestamp
    - expires_at: timestamp default now() + 30 days
    - metadata: jsonb
    - created_at: timestamp default now()
    ```
    RLS: SELECT/UPDATE where recipient_id = auth.jwt().sub AND org_id = auth.jwt().org_id. INSERT via adminDb only (system creates notifications). No DELETE (use dismissed_at soft delete per research anti-pattern guidance). Index on (recipient_id, read_at) for unread count queries. Index on (recipient_id, created_at) for chronological listing.

    **push_subscriptions** — Create `src/lib/db/schema/push-subscriptions.ts`:
    ```
    - id: uuid PK
    - user_id: uuid FK(profiles) not null
    - org_id: uuid not null
    - endpoint: text unique not null
    - p256dh: text not null
    - auth: text not null
    - device_hint: text (best-effort: 'ios' | 'android' | 'desktop')
    - created_at: timestamp
    - last_used_at: timestamp
    ```
    RLS: SELECT/INSERT/DELETE where user_id = auth.jwt().sub (users manage their own subscriptions). adminDb for server-side reads during push sends (per research Pitfall 8).

    **notification_preferences** — Create `src/lib/db/schema/notification-prefs.ts`:
    ```
    - id: uuid PK
    - user_id: uuid FK(profiles) not null
    - org_id: uuid FK(orgs) not null
    - notification_type: text not null
    - push_enabled: boolean default true
    - email_enabled: boolean default true
    - in_app_enabled: boolean default true
    - updated_at: timestamp
    UNIQUE(user_id, org_id, notification_type)
    ```
    RLS: SELECT/INSERT/UPDATE/DELETE where user_id = auth.jwt().sub AND org_id = auth.jwt().org_id. Per user decision: per-org defaults with per-user override.

    Export all three from schema index.ts. Add relations in relations.ts (notification -> profile, subscription -> profile).

    Run `drizzle-kit generate` for migration. Verify RLS policies per MEMORY.md guidance.

<task type="auto">
  <name>Task 2: Web Push actions and unified notification dispatch</name>
  <files>src/actions/push.ts, src/lib/notifications/dispatch.ts</files>
  <action>
    **Push actions** — Create `src/actions/push.ts`:
    - Install `web-push` and `@types/web-push`: `npm install web-push @types/web-push`.
    - `subscribeUserPush(subscription: PushSubscriptionJSON)`: Server action. Gets user from JWT. Upserts push_subscriptions row with endpoint/p256dh/auth. Uses withRls. Detects device_hint from request headers user-agent (best-effort: iOS Safari, Android Chrome, Desktop).
    - `unsubscribeUserPush(endpoint: string)`: Deletes the subscription row. Uses withRls.
    - `sendPushToUser(userId: string, payload: { title: string; body: string; url?: string })`: Internal function (not "use server"). Uses adminDb to query push_subscriptions for user. Calls `webpush.sendNotification()` for each. `Promise.allSettled()` to handle failures gracefully. On 410 Gone, deletes the subscription row. Configures VAPID details from env vars (NEXT_PUBLIC_VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY).
    - Set VAPID contact to `mailto:${process.env.VAPID_CONTACT_EMAIL || 'admin@poolco.app'}`.

    **Unified dispatch** — Create `src/lib/notifications/dispatch.ts`:
    - `notifyUser(recipientId: string, orgId: string, event: NotificationEvent)`: The single entry point for all notifications.
      1. Check user's notification_preferences for this event type (adminDb query). If no preference row exists, use defaults (all enabled per user decision).
      2. If `in_app_enabled`: INSERT into user_notifications with title, body, link, urgency, type.
      3. If `push_enabled`: Call `sendPushToUser(recipientId, { title, body, url: link })`.
      4. If `email_enabled` and the event type has an email template: Queue for email (fire-and-forget, non-blocking).
      5. All operations are non-blocking — failures logged but never thrown.

    - `notifyOrgRole(orgId: string, role: 'owner' | 'office' | 'tech' | 'owner+office', event: NotificationEvent)`: Convenience function that queries all profiles with the given role(s) in the org, then calls `notifyUser` for each. Used for broadcasts like "notify all office staff when a payment is received."

    - `NotificationEvent` type:
      ```
      { type: string; urgency: 'needs_action' | 'informational'; title: string; body: string; link?: string; metadata?: Record<string, unknown> }
      ```

    - Define `NOTIFICATION_TYPE_CONFIG` map: for each notification_type, define default urgency and target roles. Examples:
      - `stop_completed`: urgency='informational', roles=['owner', 'office']
      - `payment_failed`: urgency='needs_action', roles=['owner', 'office']
      - `tech_assigned`: urgency='informational', roles=['tech']
      - `weather_proposal`: urgency='needs_action', roles=['owner', 'office']
      - `chemistry_alert`: urgency='needs_action', roles=['owner', 'office']
      etc. Cover all NOTIF-05 through NOTIF-23 event types.

    Note: This plan creates the infrastructure. Plans 10-10 and 10-11 will wire the actual event triggers into existing server actions.
  </action>
