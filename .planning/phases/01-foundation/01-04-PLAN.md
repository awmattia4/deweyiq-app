    1. Create `src/lib/offline/db.ts` — Dexie IndexedDB schema:
       ```typescript
       import Dexie, { type Table } from "dexie"

       export interface SyncQueueItem {
         id?: number          // Auto-incremented
         url: string          // API endpoint
         method: string       // HTTP method
         body: string         // JSON-serialized body
         headers: Record<string, string>
         createdAt: number    // Date.now()
         retries: number      // Retry count
         status: "pending" | "processing" | "failed"
         lastError?: string   // Last error message
       }

       export interface CachedRoute {
         id: string           // stop_id or route_id
         data: unknown        // Full route/stop data (JSON-serializable)
         cachedAt: number     // When cached
         expiresAt: number    // When stale
       }

       class OfflineDB extends Dexie {
         syncQueue!: Table<SyncQueueItem>
         routeCache!: Table<CachedRoute>

         constructor() {
           super("poolco-offline")
           this.version(1).stores({
             syncQueue: "++id, createdAt, retries, status",
             routeCache: "id, cachedAt, expiresAt",
           })
         }
       }

       export const offlineDb = new OfflineDB()
       ```

    2. Create `src/lib/offline/sync.ts` — sync engine:
       - `enqueueWrite(url, method, body, headers)`: Adds a write to the sync queue. If online, immediately attempts to process. If Background Sync API is available (Android), registers a sync tag. Otherwise, relies on the `online` event listener.

       - `processSyncQueue()`: Reads all `pending` items from syncQueue ordered by createdAt. For each:
         a. Mark as `processing`
         b. Attempt fetch(url, { method, body, headers })
         c. On success (2xx): delete from queue
         d. On client error (4xx): delete from queue (won't succeed on retry), log error
         e. On server error (5xx) or network error: increment retries, set status back to `pending`, set lastError
         f. If retries >= MAX_RETRIES (Claude's discretion — suggest 5): set status to `failed`

       - Retry backoff: exponential backoff with jitter. `delay = min(baseDelay * 2^retries + random(0, 1000), maxDelay)`. Suggested: baseDelay = 1000ms, maxDelay = 60000ms. (Claude's discretion per user decision.)

       - `initSyncListener()`: Called once on app mount.
         a. Listen for `online` event on window — when fired, call `processSyncQueue()`. This is the cross-platform approach (works on both Android and iOS per research).
         b. If Background Sync API is available, register service worker sync handler as enhancement.
         c. Also attempt sync on app visibility change (`visibilitychange` event, `document.visibilityState === 'visible'`) — catches cases where the device reconnected while app was in background.

       - `getSyncQueueStatus()`: Returns `{ pending: number, failed: number, processing: number }` by querying counts from syncQueue.

       - Per user decision: "Sync failure: auto-retry silently in background; only alert user on final failure after retries exhausted."
       - Per research: "iOS does not support Background Sync API — use online event for cross-platform sync."
  </action>
