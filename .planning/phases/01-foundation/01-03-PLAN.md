<task type="auto">
  <name>Task 2: Build team invite flow and customer portal login</name>
  <files>

  <action>
    1. Create `src/actions/auth.ts` — shared server actions for auth:
       - `signOut()`: calls `supabase.auth.signOut()` and redirects to /login.
       - `getCurrentUser()`: calls `supabase.auth.getClaims()`, returns typed user with role and org_id from `app_metadata`. CRITICAL: use `getClaims()` not `getSession()`.
       - Type definition: `AuthUser = { id: string, email: string, role: 'owner' | 'office' | 'tech' | 'customer', org_id: string, full_name: string }`.

    2. Create `src/actions/invite.ts` — team member invite server action:
       - `inviteTeamMember(email: string, role: 'office' | 'tech' | 'customer')`:
         a. Verify the calling user is an owner (check their JWT claims via `getCurrentUser()`)
         b. Create a Supabase admin client using `SUPABASE_SERVICE_ROLE_KEY` (server-only, NEVER exposed to client)
         c. Call `supabaseAdmin.auth.admin.inviteUserByEmail(email, { redirectTo: \`\${process.env.NEXT_PUBLIC_APP_URL}/auth/callback?type=invite\` })`
         d. Immediately set `app_metadata` on the created user: `supabaseAdmin.auth.admin.updateUserById(user.id, { app_metadata: { role, org_id: callingUser.org_id } })`
         e. Create a profile row in the profiles table (using adminDb to bypass RLS for this admin operation): `INSERT INTO profiles (id, org_id, full_name, email, role)` with the invited user's id, the owner's org_id, email as placeholder name, and the assigned role.
         f. Return success or error.
       - Per user decision: "owner sends email invite with pre-assigned role; invitee clicks link and sets password — no approval step."
       - Per research: PKCE is NOT supported with inviteUserByEmail — the invite email contains a one-time token.
       - The invitee's callback flow: click email link -> /auth/callback exchanges token -> user is authenticated with correct role/org_id -> redirected to role landing.

    3. Create `src/app/portal/layout.tsx`:
       - Portal layout separate from staff app — per user decision: "customer portal shows the pool company's own logo and colors."
       - Phase 1: use a minimal portal layout with a different header style (lighter background, placeholder for company logo).
       - No auth guard in Phase 1 layout (portal login page must be accessible). Auth check goes in portal page components.

    4. Create `src/app/portal/login/page.tsx`:
       - Customer-facing login page at /portal/login — per user decision: "separate portal login page (/portal/login) with company branding."
       - Fields: email, password (no signup — customers are invited).
       - On submit: call `supabase.auth.signInWithPassword({ email, password })`.
       - On success: redirect to /portal (will be built in Phase 8).
       - Style differently from staff login — lighter, more customer-friendly. Placeholder for company branding.
       - No Google OAuth on portal login (staff feature only).
       - Include link to reset password.

    5. Update `proxy.ts` to handle portal routes:
       - Unauthenticated users hitting `/portal/*` (except `/portal/login`) should be redirected to `/portal/login` (NOT /login).
       - Unauthenticated users hitting `/(app)/*` routes should be redirected to `/login`.
       - Already-authenticated users hitting `/login` should be redirected to their role landing.
