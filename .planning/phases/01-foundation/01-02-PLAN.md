    1. Create `src/lib/db/index.ts` — the Drizzle client with .rls() transaction wrapper:
       ```typescript
       import { drizzle } from "drizzle-orm/postgres-js"
       import postgres from "postgres"
       import { sql } from "drizzle-orm"
       import * as schema from "./schema"

       // CRITICAL: prepare: false required for Supabase transaction-mode pooler
       const client = postgres(process.env.DATABASE_URL!, { prepare: false })
       export const adminDb = drizzle({ client, schema })

       type SupabaseToken = {
         sub: string
         role?: string
         user_role?: string
         org_id?: string
       }

       // Use this for ALL user-facing queries — sets RLS context per transaction
       export function createRlsClient(token: SupabaseToken) {
         return adminDb.transaction(async (tx) => {
           await tx.execute(sql`
             SELECT set_config('request.jwt.claims', ${JSON.stringify(token)}, TRUE),
                    set_config('request.jwt.claim.sub', ${token.sub}, TRUE),
                    set_config('request.jwt.claim.role', ${token.user_role ?? token.role ?? 'authenticated'}, TRUE),
                    set_config('request.jwt.claim.org_id', ${token.org_id ?? ''}, TRUE)
           `)
           await tx.execute(sql`SET LOCAL ROLE authenticated`)
           return tx
         })
       }
       ```

       CRITICAL: This wrapper must be used for EVERY user-facing query. `adminDb` bypasses RLS and must only be used for admin operations (invite creation, webhook handlers, migrations).

    2. Create `supabase/custom-access-token-hook.sql`:
       - The Postgres function `public.custom_access_token_hook(event jsonb)` that promotes `org_id` and `role` from `app_metadata` into top-level JWT claims.
       - Use the exact pattern from the research:
         - Read `org_id` and `role` from `event -> 'claims' -> 'app_metadata'`
         - Set them as top-level claims: `org_id` and `user_role` (use `user_role` to avoid conflicting with Supabase's built-in `role` claim which is "authenticated")
         - Grant execute to `supabase_auth_admin`
         - Revoke execute from `authenticated`, `anon`, `public`
       - Add a comment at the top: "Run this SQL in Supabase Dashboard > SQL Editor. Then enable the hook in Dashboard > Authentication > Hooks > Custom Access Token."

    3. Create `supabase/org-creation-trigger.sql`:

         - Sets `app_metadata` on the new user with `org_id` and `role` using `auth.admin.updateUserById` or direct update to `auth.users.raw_app_meta_data`
       - This trigger only fires for NEW signups (not invites — invites already have org_id/role set via admin API)
       - Add guard: `IF NEW.raw_app_meta_data ->> 'org_id' IS NULL THEN` (skip if org_id already set, meaning this is an invited user)
       - Add a comment: "Run this SQL in Supabase Dashboard > SQL Editor after the orgs and profiles tables exist (after running Drizzle migrations)."
