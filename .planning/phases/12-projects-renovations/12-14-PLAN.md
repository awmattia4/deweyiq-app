**src/actions/projects-billing.ts:**

**Progress invoices (PROJ-62, 63):**
- `onPhaseComplete(token, phaseId)` — Called when a phase is marked complete (integrate with completePhase from Plan 12). Check if any payment_milestones have trigger_phase_id matching this phase AND invoice_id is NULL. For each matched milestone:
  - Calculate invoice amount: milestone.amount - (milestone.amount * project.retainage_pct / 100)
  - Calculate retainage_held: milestone.amount * project.retainage_pct / 100
  - Create DRAFT invoice (status='draft') in the EXISTING invoices table with: project_id, invoice_type='project_progress', project_milestone_id, retainage_held, total=computed amount. Use the same invoice numbering sequence (PROJ-66 -- no separate counter per research Pitfall 5).
  - Update milestone.invoice_id and milestone.status='invoiced'.
  - Create alert for office: "Progress invoice ready for review."
  - Per user decision: "auto-generated on phase completion and held for review -- system creates draft invoice, notifies office, office must explicitly send."
- `generateProgressInvoice(token, milestoneId)` — Manually trigger invoice generation for a milestone (in case auto-generation is bypassed).

**Retainage (PROJ-64):**
- `getRetainageSummary(token, projectId)` — Query all non-void project invoices: sum(retainage_held) = total held. CRITICAL: Calculate fresh from invoice records, NOT from a stored running total (per research Pitfall 8 -- voided invoices affect the balance).
- Retainage info displayed on each progress invoice line.

**Final invoice (PROJ-65):**
- `generateFinalInvoice(token, projectId)` — Create the final invoice including:
  1. Remaining contract balance (contract_amount - sum of prior invoice totals)
  2. Retainage release: retainage_released = sum of all retainage_held from prior invoices
  3. Any outstanding CO amounts (from 'collect_immediately' COs not yet invoiced)
  4. invoice_type='project_final', retainage_released set.
  - This should only be called after final walkthrough sign-off (Plan 15 integrates this trigger).

**Profitability (PROJ-67):**
- `getProjectProfitability(token, projectId)` — Compute:
  - Revenue: contract_amount (original + approved COs)
  - Costs: sum(material actual costs) + sum(labor hours * hourly rate from profiles) + sum(sub payments) + sum(permit fees) + equipment costs
  - Margin: (revenue - costs) / revenue * 100
  - Projected margin: if project is in progress, project remaining costs based on % complete
  - Return: { revenue, totalCosts, materialCosts, laborCosts, subCosts, permitCosts, margin, projectedMargin, isAtRisk }
- `checkProfitabilityAlerts(token, projectId)` — Compare margin against org_settings thresholds (margin_floor_pct, overrun_alert_pct). If actual margin < margin_floor OR actual costs > budget * (1 + overrun_alert_pct/100), create alert. Per user decision: "configurable threshold per company."

Add `project_margin_floor_pct` (numeric, default 15) and `project_overrun_alert_pct` (numeric, default 20) to org_settings if not already present.

**Cancellation (PROJ-90, 68):**
- `calculateCancellationSettlement(token, projectId)` — Compute:
  - Completed work value (invoiced and paid amounts)
  - Non-returnable materials (ordered but can't return)
  - Cancellation fee (from project.cancellation_policy JSONB if set)
  - Deposit received - completed work - materials - fee = refund (or balance owed)
  - Return settlement breakdown for office review.
- `recordCancellationRefund(token, projectId, refundData)` — Record refund via existing payment infrastructure. Update project status='cancelled'.

**Suspension (PROJ-92):**
- `suspendProject(token, projectId)` — Triggered by overdue project invoices. Set status='suspended'. Stop all work phases. Notify customer and office. Configurable cure period: `project_cure_period_days` in org_settings (default 14 days).
- `checkSuspensionTriggers()` — Scheduled: find projects with project invoices overdue > X days. Auto-suspend if not already.
