**src/components/projects/signature-pad.tsx:**
Per user decision: "E-signature uses draw/type signature (canvas to draw or typed stylized) -- produces a professional-looking signed PDF."

Two modes:
1. **Draw mode**: react-signature-canvas with pen color #0f172a on white background. Clear button, undo last stroke. Mobile-friendly touch handling (signature_pad handles this automatically).
2. **Type mode**: Text input with a stylized cursive font rendering (use a Google font like 'Dancing Script' or similar). Preview of how it will look. Convert rendered text to data URL for PDF embedding.

Both modes output a PNG data URL via `onSign(dataUrl)` callback. The component also captures signed_name (text input) and captures the current timestamp and client IP (for legal compliance).

**src/actions/projects-approval.ts:**
- `getProposalPublicData(proposalId)` -- Called with adminDb (no RLS -- customer has no auth session). Fetches proposal with tiers, line items, addons, payment milestones, project info, company branding (org logo, name, colors). Returns everything needed to render the approval page.
- `approveProposal(proposalId, data)` -- data includes: selected_tier, selected_addon_ids[], signature_data_url, signed_name, signed_ip, signed_at. Uses adminDb. Updates proposal: status='approved', selected_tier, signature_data_url, signed_name, signed_ip, approved_at=now(). Marks selected addons as is_selected=true. Updates project: stage='proposal_approved', contract_amount=(tier price + selected addons total). Creates payment milestones from schedule if not already created. Creates activity_log entry. **CRITICAL (PROJ-28 trigger): After setting stage='proposal_approved', call `await populateMaterialsFromProposal(token, projectId)` (imported from `@/actions/projects-materials`) to seed the project material list from the approved proposal's line items.** This is the bridge between proposal approval and material procurement -- without this call, Plan 12-09's material list would be empty. Returns { proposalId, projectId, depositAmount, depositMilestoneId }.
- `submitChangeRequest(proposalId, customerNotes)` -- Uses adminDb. Insert into proposal_change_requests. Create alert for office. Send notification email to office.
- `recordOfflineDeposit(proposalId, paymentData)` -- For cash/check deposits. Uses adminDb. Record payment, update milestone status, advance project stage to 'deposit_received'.
- `sendDepositReminder(projectId)` -- Send reminder email for pending deposit. Called by a scheduled job (daily scan for projects in 'proposal_approved' stage with no deposit after X days).

**src/app/api/projects/deposit/route.ts:**
POST endpoint that creates a Stripe PaymentIntent for the deposit amount. Mirrors the existing `/api/payments/create-intent/route.ts` pattern. Uses the org's connected Stripe account. Returns clientSecret for the PaymentElement. On payment success (via webhook or client confirmation): update milestone status='paid', advance project stage to 'deposit_received', add activity_log entry.

Split deposit support (PROJ-20): If the customer selects "split deposit," create two payment milestones -- first half due now, second half auto-scheduled (e.g., 7 days later). The first payment is collected inline; the second generates a payment link sent via email.
